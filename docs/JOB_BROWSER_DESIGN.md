# 작업 브라우저 설계 — 패싯 탐색 카드 브라우저 (토의 박제)

작성 2026-07-13. **상태: 토의 완결 · 구현 미착수.** 홈 대시보드의 작업 카드 목록이
작업 수 증가에 스캔 불능이 되는 설계 결함에 대한 토의를 박제한다. 구현 착수 시
`feature` 워크플로(plan 모드)로 영향도를 뜨고, 이 문서가 요건 원장이 된다.

> **문제:** 작업이 "적격심사 - 1억미만 / 적격심사 - 고시이상 / 중기간제품 - 1억미만 …"
> 처럼 늘면 카드 평면 리스트가 한눈에 안 들어온다. 접두어가 N번 반복되며 시각
> 노이즈가 되고, 눈이 매번 문자열을 카테고리×구간으로 재분해해야 한다.

---

## 1. 진단 — 문제는 카드가 아니라 모델이다

- `Job.name` 은 평문 문자열(`core/job.py:94`)이고, 정렬은 이름순 고정
  (`JobRegistry.list_jobs`, `job.py:217`), 검색·필터·그룹핑 부재(`gui/home.py`).
- 작업을 1급으로 승격시키면서 **사람이 인지하는 업무 택소노미도 함께 들어왔는데**,
  그 좌표가 별도 구조 없이 `name` 하나에 접혀 있다. `name` 이 ① 기술적 정체성
  (durable 바인딩 {템플릿, 매핑, 파일명}의 이름) ② 업무적 좌표(택소노미상 위치)를
  **동시에 짊어지는(double duty)** 눌린 설계 — 문헌이 명시적으로 경고하는
  안티패턴이다(§6 프라이어 아트, [Nayuki]).
- 화면의 스캔 불능은 이 모델 눌림의 그림자일 뿐이므로, "카드냐 표냐"는 두 번째
  질문이다. 첫 번째 질문은 **계층을 진짜 구조로 승격시킬 것인가**였다.

## 2. 도메인 배경 — 조합 폭발과 조립의 예감

- 공고서 = "법정 필수 코어(민법 바닥) + 재량 블록". 필수만 지키면 어떤 내용도
  들어갈 수 있음 — 이상 사분면(반복성×법적효력)의 성질 그대로.
- 현 구조는 코어×블록 선택의 곱집합을 **템플릿 파일 N개로 미리 열거**하고 작업이
  그중 하나를 가리키는 방식. 카드 폭발은 "생성해야 할 공간을 손으로 열거하고
  있다"는 UI의 비명이며, 종착지는 **조건부 문서 조립**이다.
- **단 조립 엔진은 지금 짓지 않는다**(재발명 리트머스 미충족 — 담당자별 실질
  수렴 중이면 열거로 족함). 대신 **조건 축을 명시 구조로 뽑는 것**이 두 미래
  (열거 유지 / 조립 도래) 모두에서 옳은 유일한 no-regret 수: 지금은 그룹핑
  기반이 되고, 조립이 오면 그대로 조건 어휘가 된다.
- 식별된 축(도메인 지식으로서 — 코드에 박는 목록이 아님): 목적물 구분
  (물품/용역/공사 — 당장은 물품만), 목적물별 금액구간(법적 임계 — 심사 방식이
  갈림), 낙찰자선정방법, 중기간경쟁제품 여부(낙찰방법과 기타에 **동시 관여**하는
  cross-cutting facet — 트리에 못 접는다는 결정적 신호). 템플릿 변형은 leaf
  아래로 더 창발 가능(→ 미래 조립의 "손으로 만든 열거형", 지금은 정상).

## 3. 추상화 — 탐색이란 무엇인가

> **탐색 = 후보 집합을 연속된 제약으로 좁혀 들어가는 것.** 사용자의 각 조작
> (가지 선택·칩·토글)은 전부 "한 차원에 값을 못박는 술어" 하나이고, 계층 탐색은
> 부분 할당(차원→값)을 점증적으로 쌓는 conjunctive query다. GUI상 의도선언 =
> 그 술어 하나하나.

이 시선에서 **축/깊이/태그/토글은 경쟁 선택지가 아니라 같은 원시연산의 표현
형태**이고, 차이는 딱 하나 — **순서를 고정하느냐**:

- **트리(축·깊이)** = 차원들에 순서를 미리 못박고 좁힘을 중첩 containment 로
  그린 것. 깊이 = 순서에 편입시킨 차원 수. 전수 통과 강제.
- **facet(칩·토글)** = 순서 없이 아무 순서로나 제약을 얹게 둔 것. 선택적.
- **트리는 "facet 에 순서를 얼린 특수 케이스"** — facet 이 일반형이다.

후보로 거론된 세 시스템(카드+정렬+태그 / 계층화 카드+DnD / search)도 한 시스템의
세 동사로 수렴: **Group**(차원을 containment 축으로 → 섹션), **Filter**(차원에
값 선택 → 칩), **Locate**(자유텍스트 술어 → 검색). 셋 다 같은 태그 데이터 위에서
돈다. 시스템은 하나 — **사용자가 group-by 를 제어하는 faceted 카드 브라우저.**

## 4. 결정 (ADR)

| # | 결정 | 근거 | 기각한 대안 |
|---|---|---|---|
| D1 | **분류를 명시 구조로 승격**: 작업에 분류 태그를 실 데이터로 부여. 이름 문자열 파싱으로 그룹핑하지 않는다 | 이름 ` - ` 파싱은 조용한 추측(confirm-or-alarm 위반). 명시 태그는 그룹핑·필터를 견고하게 하고 미래 조립의 조건 어휘가 됨 | 렌더타임 이름 파싱 — 구분자 없는 이름의 조용한 오분류 |
| D2 | **태그는 차원-불가지**: 작업은 `{축이름 → 값}` 얇은 매핑만 든다. 코드는 "물품"·"적격심사"가 뭔지 모른다. 타입 필드(enum/bool) 발명 금지 | "도메인 지식을 그대로 구조에 넣는 게 아니라 **도메인 지식을 감당할 구조**면 족하다"(토의 중 3회 적발된 원칙). 택소노미 변경 시 코드 무변경 | `category: enum{물품,용역,공사}` 등 타입 필드 — 도메인을 코드에 박음 |
| D3 | **차원은 발견**: 사용 가능한 축 목록은 작업들에 실제로 붙은 태그에서 발견한다. authored 축 레지스트리 없음 | 축 추가가 데이터 행위가 됨(코드·설정 무변경) | 축 목록 config — 도메인 재유입 통로 |
| D4 | **렌즈는 사용자 소유 상태**: 지속되는 "차원 키 순서 리스트"(group-by 시퀀스) 하나. 리스트에 든 차원이 순서대로 트리 depth, **안 든 나머지는 자동으로 facet 칩/토글**. 깊이 = 사용자가 정한 리스트 길이 | 어느 차원이 트리를 이끄는가 자체가 도메인 정책 — 개발자가 못박으면 D2 위반의 재발. 사용자가 노코드로 재그룹핑 | 분석가 authored 기본 렌즈 고정 — "금액구간이냐 낙찰방법이냐"를 우리가 맞혀야 하는 부담 |
| D5 | **기본값은 씨앗일 뿐**: 초기 group-by 는 금액구간 계열로 씨딩하되, 틀려도 싼 값 — 사용자가 런타임에 즉시 교체 | D4 가 결정을 증발시킴. 씨앗은 첫 화면 품질용 | 기본값 선정에 조사·확정 게이트 — 과잉 |
| D6 | **분류는 job 의 속성(attribute), 1급 객체 아님**: 택소노미 트리는 태그에서 창발. 트리 실체(빈 카테고리·카테고리 메타·중앙 개명)는 발명하지 않음 | 속성 스탠스는 1급 승격 경로를 막지 않음(태그가 그대로 마이그레이션 소스). 선불 관계 관리 부담 회피 | 택소노미 1급 엔티티 — 실증 안 된 부담 선불 |
| D7 | **기존 작업 마이그레이션은 제안-확정**: 이름 접두어에서 태그 기본값을 **제안**하고 사용자가 확정/수정. 무확정 자동 태깅 금지 | 파싱을 "숨은 추측"이 아니라 "사용자가 확인하는 제안"으로 — confirm-or-alarm 정합 | 일괄 자동 태깅(조용) · 전건 수동 재입력(과잉) |
| D8 | **UI = 그룹 섹션 + facet 칩 + 카드 유지**: group-by 차원 값별 접이식 섹션 헤더(건수 표시), 나머지 차원은 상단 칩/토글, 카드 스펙(`style.py` 카드 언어) 불변 | 접두어 노이즈가 헤더 한 줄로 흡수. 디자인 언어 훼손 최소 | 밀집 테이블(카드 언어와 이질) · 매트릭스 표(축 정규화 선행 필요, 범위 과대) · 마스터-디테일(현행 좌우 투트랙과 충돌) |
| D9 | **카드 손배치(DnD) 금지**: 계층은 태그에서 파생. DnD 를 쓴다면 렌즈(그룹핑 순서 재배열)에만 | 위치 손배치는 태그 정보의 위치 중복 — "구조를 항목마다 재박기"로 회귀 | 카드 드래그 배치 |
| D10 | **패싯 표준 의미론 채택**: ① 모든 섹션·facet 값에 건수 표시 ② 0건으로 가는 제약은 회색/숨김(빈 결과 회피) ③ facet 내 OR / facet 간 AND ④ 활성 제약 칩 상시 노출 + 개별 해제 | Flamenco 설계 목표·이커머스 표준(§6) — 통제감 유지, 막다른 길 차단 | 자체 의미론 발명 |
| D11 | **값 순서는 지연 가능한 편집 데이터**: 순서형 차원(1억미만<고시미만<고시이상)의 값 순서는 코드가 아니라 차원별 편집 가능 데이터. v1 은 삽입순 허용, 거슬릴 때 도입 | 알파벳 정렬로 못 맞추는 유일한 진짜 도메인 조각 — 데이터로 격리 | 순서를 코드 상수로 · v1 필수화 |

## 5. 족하다 컷 — v1 범위와 보류

**v1 (족하다):** 작업 태그(D2) + 차원 발견(D3) + **group-by 1단**(D4) + facet
칩/토글 + 섹션 접기 + 건수/0억제/OR·AND/활성칩(D10) + 마이그레이션 제안 흐름(D7).

**보류 (밥값 대기 — 승급 신호 명시):**

| 항목 | 승급 신호 |
|---|---|
| 조건부 문서 조립 엔진 | **리트머스: "블록 하나 바꾸려고 템플릿 통째 복붙 → near-duplicate 생성"이 담당자들 사이에서 반복 관찰될 때.** 그 전까지 열거+그룹핑이 정직한 답 |
| 검색(Locate) | 작업 수가 커져 group+facet 으로도 느릴 때. 검색은 recall(이름을 알아야 함), 현 단계는 recognition 우선. 현행 ~11건은 group+facet 으로 완전히 풀림 |
| 다단 group-by(sub-grouping) | 1단으로 섹션이 여전히 비대할 때 (Notion sub-group 이 검증한 확장로) |
| 저장된 이름붙은 뷰 | 렌즈 교체가 잦아 왕복 비용이 실증될 때 (Notion/Airtable views) |
| facet 내 검색·상위N+더보기 | 한 차원의 값이 수십 개로 늘 때 |
| 값 순서 편집 데이터(D11) | 삽입순이 실제로 거슬릴 때 |
| 택소노미 1급 승격(D6 역전) | 빈 카테고리·카테고리 메타·중앙 개명이 실제로 필요해질 때 — 태그가 마이그레이션 소스 |
| 낙찰방법의 종속 규칙(구간→심사방식 결정) | 조립 엔진과 동시 — 그 전까지 자유 facet 으로 기록만 |

## 6. 프라이어 아트 — 이 문제는 이름이 있다

토의로 독립 재구성한 구조가 **faceted navigation** 과 수렴함을 확인했다.
방향 리스크를 크게 낮추는 근거:

- **Marti Hearst / Flamenco(UC Berkeley, 13년 연구)** — 설계 목표가 본 토의
  결론과 일치: 유연한 내비게이션 · browse/search 통합 · refine/expand 유동 교대 ·
  **빈 결과집합 회피** · 사용자 통제감 유지.
  [설계 권고](https://flamenco.berkeley.edu/papers/faceted-workshop06.pdf) ·
  [Search UIs Ch.8](https://searchuserinterfaces.com/book/sui_ch8_navigation_and_search.html)
- **Faceted classification** — "패싯 = 직교 범주 집합, 고정 택소노미와 달리
  facet 들의 여러 순서로 다중 경로 탐색" = §3 의 "트리는 순서를 얼린 특수
  케이스". [Wikipedia](https://en.wikipedia.org/wiki/Faceted_classification)
- **[Nayuki, tags-not-folders](https://www.nayuki.io/page/designing-better-file-organization-around-tags-not-hierarchies)**
  — 경로/이름의 double-duty(기계 검색+인간 분류) 지적 = §1 진단의 축자적 확인.
  구분자로 계층을 이름에 인코딩하는 것은 문헌이 지목하는 교과서 안티패턴.
- **Notion·Airtable** — "사용자 소유 group-by 렌즈"(D4)의 상용 선례: Notion 은
  Select 속성 group-by+sub-grouping, Airtable 은 그룹 순서 드래그 재배열·빈 그룹
  숨김·그룹별 카운트. §5 보류 항목들의 확장로가 이미 닦여 있음을 보증.
  [Airtable](https://support.airtable.com/docs/grouping-records-in-airtable) ·
  [Notion](https://www.notion.com/help/boards)
- **이커머스 패싯 표준** — facet 내 OR/간 AND · 건수 표시 · 0건 차단 · "이 속성이
  facet 자격이 있나 = 사용 빈도 테스트"(D10, 그리고 축 선정을 경험 문제로 두는
  D3~D5 의 방증).
  [Broken Rubik](https://www.brokenrubik.com/blog/faceted-search-best-practices) ·
  [A List Apart](https://alistapart.com/article/design-patterns-faceted-navigation/)
