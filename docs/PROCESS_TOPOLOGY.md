# 프로세스 topology와 작업 저장소 소유권

HWPX Filler가 지원하는 작업 저장 topology는 **한 디렉터리당 writer 프로세스 하나**다.
네트워크 파일시스템의 분산 쓰기나 여러 앱 프로세스의 동시 편집은 지원하지 않는다.

## 제품 경계

- 웹앱은 `HWPXFILLER_HOME` 기준 단일 인스턴스 뮤텍스를 `WebFrontend` 생성보다 먼저 잡는다.
  따라서 기본 `jobs` 디렉터리를 쓰는 앱 프로세스는 하나다.
- CLI는 사용자가 지정한 원본·출력 경로를 직접 처리하며 `JobRegistry`를 열지 않는다.
- `--selftest`는 서로 다른 임시 홈에서 순차 실행하는 테스트 하네스라 앱 뮤텍스를 우회한다.

## 같은 프로세스

같은 디렉터리를 가리키는 여러 `JobRegistry` 인스턴스는 경로 단위 `RLock`을 공유한다.
`write_lock()`은 외부의 read→modify→save 구간까지 같은 임계구역에 넣는다. 따라서 화면이나
컨트롤러가 registry 인스턴스를 따로 만들더라도 stale read로 상대 변경을 되돌리지 않는다.

## 다른 프로세스

첫 쓰기에서 해당 디렉터리의 프로세스 writer 소유권을 잡는다. 다른 프로세스가 이미 소유한
디렉터리에 쓰려고 하면 작업 JSON을 만들거나 바꾸기 전에 `JobRegistryOwnershipError`로
실패한다. 기존 앱을 닫고 다시 시도해야 한다. 프로세스 종료 시 OS가 소유권을 회수하므로
충돌 뒤 잠금 파일을 수동 삭제하는 복구 절차는 없다.

다른 프로세스의 동시 읽기는 writer의 여러 파일 일괄 변경에 대한 snapshot 일관성을 보장하지
않는다. 진단 목적 외에는 실행 중인 앱의 작업 디렉터리를 별도 프로세스에서 읽지 않는다.
