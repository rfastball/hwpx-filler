## 1. 데이터 입력 어댑터 층 — UX 보조

**문제**: `DataSource`(`data/base.py`)는 `records()`+`fields()`뿐이라 템플릿
필드명(`진행상태`, `계약명` 등)과 데이터 컬럼 헤더가 **정확히 일치해야만** 채워진다.
현실에선 거의 안 맞는다. 원본 VBA는 이걸 `frmErpPreview`+`modFuzzyMatch`
(Unknown 항목 수동 매핑 HITL)로 이미 풀었다 — 그 아이디어를 되살린다.

**제안: `DataSource`와 `Engine` 사이에 매핑 계층**
```
템플릿필드  ←  Mapping  →  { 소스컬럼 | 상수 | 변환식 }
```
- **퍼지 자동 제안**: 템플릿 필드 vs 데이터 컬럼 유사도 추천(한글 자모 정규화 +
  부분일치). 사용자는 확인/수정만.
- **매핑 프로파일 저장**(템플릿별 JSON): 같은 레이아웃이면 다음부터 자동 적용.
  → **ERP 연동의 결정적 이득**. ERP 필드코드(`CNTRCT_NM`)는 템플릿 한글명과
  안 맞으니, "코드→템플릿명" 매핑을 프로파일로 고정하면 `ErpApiDataSource`로
  갈아끼워도 매핑 재사용.
- **변환/서식 계층**: 날짜·금액 포맷팅(원본 `modDataHydrator`/`modFormatUtils`의
  WYSIWYG 역할). 매핑 항목별 optional transform.
- **데이터 소스 마법사**: 헤더 행 자동 감지, 시트 선택, CSV 인코딩 감지
  (`ExcelDataSource`의 `header_row`/`sheet` 인자를 UI로 노출).

의존: 템플릿 스키마(축)가 필드 목록+타입을 주면 매핑 UI 좌측 열이 자동 생성.

## 2. 메일머지 기능 및 UX 고도화

메일머지 본질(1 템플릿 × N 레코드 → N 문서)은 이미 `batch.py`가 수행. 고도화는
UX와 출력 제어.

- **위저드(스텝) 흐름**: ① 템플릿 → ② 데이터 연결 → ③ 필드 매핑(1번) →
  ④ 미리보기 & 행 선택 → ⑤ 생성.
- **행 선택 체크박스**: 원본 `modUIUtils`(전체선택/해제) 역할. 생성 대상만 선택.
- **레코드별 실시간 미리보기**: 채워질/빈 필드 표시 — `validate.py` 판정 로직 재사용.
- **파일명 규칙 강화**: `naming.py`의 `{{key}}` 치환에 날짜 토큰·일련번호·
  **충돌 시 자동 `_1/_2` 접미사** 추가.
- **단일 문서 병합 옵션**: N레코드를 문서 1개에 N섹션/페이지로 이어붙이기.
  파서 심화 필요(섹션 본문 복제·OCF 재구성) → 3번과 연결.
- **PDF 내보내기(한계 명시)**: HWPX→PDF는 한글(HWP) 설치 또는 별도 변환기 필요.
  순수 Python 불가 — 필요 시 "한글 COM 자동화 어댑터"를 옵션 모듈로 분리 설계.

## 3. 파서 고도화로 얻을 향후 기능들

계약·구매 업무 가치 순:

- **① 반복 영역/표 채우기 (최대 가치)**: 품목 리스트처럼 **행 수 가변 표**를
  데이터로 채우기. `hp:tbl`/`hp:tr`/`hp:tc` 이해 + 행 복제. 구매·계약 문서의
  핵심(품목·수량·단가). 사실상 "메일머지 2.0".
- **② 템플릿 스키마 추출/내보내기 (토대)**: `required_fields()`를 확장해
  필드 타입·표 영역·위치까지 JSON 스키마로 출력 → **입력 폼 자동 생성**(1번 매핑
  UI 자동 구축), 검증 강화(미치환 `{{}}` 잔존 검사).
- **③ 리치 값 주입**: 체크박스 on/off, 여러 줄, 서식 있는 런.
- **④ 이미지/직인·로고 삽입**: `BinData` + `manifest.xml` 조작. 복잡하나 공문 가치 큼.
- **⑤ 출력 검증**: 생성물의 빈 필드/미치환 토큰 자동 확인, 부분 실패 정직 보고.

### 3-1. HWPX 규격서 개정 diff — ★ 1급 기능 (핵심 활용처)

**동기**: 규격서·표준서식은 **해마다 개정**된다. 무엇이 어떻게 바뀌었는지 사람이
수작업으로 대조하는 건 느리고 실수 잦다. 두 판본을 넣으면 변경점을 **사람이 훑기
쉽게** 보여주는 도구. → 생성기와 대등한 1급 기능. 독립 "규격서 개정 리뷰 도구"
상품성도 있음.

**성격 재정의**: 이건 채워진 문서의 **필드값 비교(L3)가 아니라 본문·조항·표의 내용
비교**다. 규격서 개정은 누름틀이 아니라 조항 문구·기준 수치·표가 바뀐다. 무게중심:
- **L2 본문 텍스트 diff = 주역**, **표 diff = 필수**(기준표·요율표)
- L1 컨테이너(엔트리 추가/삭제, 해시 비교) = 보조 신호
- L3 필드값 = 부차

**함정**: HWPX XML엔 랜덤 ID가 가득(`id="2073595120"`, `fieldid="627272811"`).
원시 diff는 ID·공백·속성순서 잡음에 파묻힘 → **의미 기반 diff** 필수.

**진짜 어려운 지점 (설계의 본체)**:
1. **문단 텍스트 재구성**: HWPX는 한 문단(`hp:p`)을 여러 `hp:t` 조각으로 쪼갠다
   (주입 로직이 파편 텍스트를 지우던 이유). 조각을 순서대로 이어 붙이고 줄바꿈·탭을
   살려 "깨끗한 문단 문자열"을 복원해야 함.
2. **내용 기반 정렬(alignment)**: 판본 간 안정 ID가 없어 문단을 위치가 아니라
   내용 유사도로 짝지어야 함(조항 삽입 시 뒤가 다 밀리므로 단순 라인 diff 부적합).
3. **문단 내 단어 단위 강조**: "요율 3%→3.5%", "1억→2억" 미세 변경을 문장 안에서
   하이라이트(difflib 토큰 diff).
4. **표 대조**: 표를 제목/위치로 매칭 후 셀 단위 비교.
5. **이동 감지**(선택): 이동한 절을 삭제+추가로 오인 안 하기.

**리뷰어 킬러 기능 — 변경 항목 추출**: 검토자가 가장 신경 쓰는 건 바뀐 **숫자**
(기준금액·요율·기한·수치)와 **조항 신설/삭제**. 원시 diff 대신 이걸 우선순위
리스트로: `3.2조 기준금액 1억→2억`, `제5조 신설`, `별표2 3행 단가 변경`.

**사람이 쓰기 좋은 출력**:
- **색상 HTML 리뷰 리포트**: 추가/삭제/수정 색분류 + 문단 내 단어 강조 + 섹션 접기
- **변경 요약**: "문단 12 추가 / 5 삭제 / 8 수정, 표 2개 변경"
- **변경 항목 리스트**(위 킬러 기능)

**재사용/신규**: `HwpxPackage`(컨테이너)·XML 순회는 재사용. 파서에 **본문 텍스트
추출기**(문단·표 단위 정규화)를 신설 — 필드 주입과는 독립된 별개 능력이자
L2 diff·향후 검증·병합의 공용 토대.
