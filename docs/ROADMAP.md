- **표시형(format) 선언 — 매핑 위저드의 WYSIWYG + 교체 가능한 서식 엔진**
  (`core/format_engine.py` + `FieldMapping.fmt`(서식 코드) + 매핑 테이블 '표시형' 열).
  **타입 논쟁의 착지점**: 타입은 소유하지 않는다 — 휴리스틱이 *추천*(스키마 배지)하고, 사람이
  매핑 시점에 **변환된 출력을 실 레코드로 보며(WYSIWYG 미리보기+스텝퍼)** 표시형을 고르고,
  프로파일에 **기억**(JSON `fmt`, 하위호환)하고 편집한다.
  - **어댑터 층**(`format_engine.FormatEngine` 프로토콜 + `ENGINE` 교체점): 서식 해석기를
    갈아끼울 수 있다. 지금은 **stdlib**(금액=`str.format` 스펙 `{:,}원`·`{:,.2f}`, 날짜=`strftime`
    `%Y-%m-%d`) — 의존성 0. **날먹 경로**로 표현력이 프리셋에 안 갇힘. babel(CLDR/엑셀 패턴)은
    후일 `ENGINE` 한 줄 교체로 가능(단 우리 축과 강점 안 겹쳐 보류 — 로케일 통화 ₩≠원 등).
  - **fmt = 서식 코드**("" = 기본 원/한글). 프리셋(원·숫자·소수·ISO·점·연월)은 **분류해 클릭하는
    씨앗**일 뿐, 고급 사용자는 '직접 입력…'으로 코드 자유 편집(Excel 갤러리+커스텀). 파싱→서식→
    실패 시 원본(degrade) — 타입 *주장*이 아니라 관대한 *시도*, WYSIWYG가 안전망.
  - 표시형 변경은 확정 해제(사람 눈 재확인).

값을 몰래 재서식하거나 "반복 표처럼 보이니" 알아서 행 복제하는 것은 불쾌선을 넘는다.

**본질/스코프 — 값 치환의 확장이지 문서 조립이 아님.** 조건부 조립(if-then 섹션),
문서 레벨 서식 조립, 단일 문서 병합(아래 2번)은 **명시적 non-goal**. 값 변환(날짜/금액
서식)·표 행복제는 경계선상 — 위 명시성 원칙으로 판정(명시 선언+미리보기면 허용).

**동결(후순위, 파킹): 법령 개정 오라클 코퍼스.** 골든(변경탐지)·합성(자작정답)을

조각). 트랙 B 이후 재개. 필요 파일 3종(law.go.kr, 개정폭 작은 법령): ① 개정 전 HWPX

가장 가까움). 절차: ⓐ 추출기가 법령 구조(조/항/호/목·별표·부칙) 온전 처리하는지
ledger 확인 → ⓑ 쌍 코퍼스 편입 → ⓒ 신구조문대비표를 정답 변경목록으로 인코딩 →
ⓓ diff 재현율/정밀도 테스트를 격리 반복으로 구축. granularity 주의: 개정문(조문

## 1. 데이터 입력 어댑터 층 — UX 보조

**문제**: `DataSource`(`data/base.py`)는 `records()`+`fields()`뿐이라 템플릿
필드명(`진행상태`, `계약명` 등)과 데이터 컬럼 헤더가 **정확히 일치해야만** 채워진다.
현실에선 거의 안 맞는다. 원본 VBA는 이걸 `frmErpPreview`+`modFuzzyMatch`
(Unknown 항목 수동 매핑 HITL)로 이미 풀었다 — 그 아이디어를 되살린다.

**제안: `DataSource`와 `Engine` 사이에 매핑 계층**
```
템플릿필드  ←  Mapping  →  { 소스컬럼 | 상수 | 변환식 }
```
- **퍼지 자동 제안**: 템플릿 필드 vs 데이터 컬럼 유사도 추천(한글 자모 정규화 +
  부분일치). 사용자는 확인/수정만.
- **매핑 프로파일 저장**(템플릿별 JSON): 같은 레이아웃이면 다음부터 자동 적용.
  → **ERP 연동의 결정적 이득**. ERP 필드코드(`CNTRCT_NM`)는 템플릿 한글명과

- **변환/서식 계층**: 날짜·금액 포맷팅(원본 `modDataHydrator`/`modFormatUtils`의
  WYSIWYG 역할). 매핑 항목별 optional transform.
- **데이터 소스 마법사**: 헤더 행 자동 감지, 시트 선택, CSV 인코딩 감지

### 3-1. HWPX 규격서 개정 diff — ★ 1급 기능 (핵심 활용처)

**동기**: 규격서·표준서식은 **해마다 개정**된다. 무엇이 어떻게 바뀌었는지 사람이
수작업으로 대조하는 건 느리고 실수 잦다. 두 판본을 넣으면 변경점을 **사람이 훑기
쉽게** 보여주는 도구. → 생성기와 대등한 1급 기능. 독립 "규격서 개정 리뷰 도구"
상품성도 있음.

**성격 재정의**: 이건 채워진 문서의 **필드값 비교(L3)가 아니라 본문·조항·표의 내용
비교**다. 규격서 개정은 누름틀이 아니라 조항 문구·기준 수치·표가 바뀐다. 무게중심:
- **L2 본문 텍스트 diff = 주역**, **표 diff = 필수**(기준표·요율표)
- L1 컨테이너(엔트리 추가/삭제, 해시 비교) = 보조 신호
- L3 필드값 = 부차

**함정**: HWPX XML엔 랜덤 ID가 가득(`id="2073595120"`, `fieldid="627272811"`).
원시 diff는 ID·공백·속성순서 잡음에 파묻힘 → **의미 기반 diff** 필수.

**진짜 어려운 지점 (설계의 본체)**:
1. **문단 텍스트 재구성**: HWPX는 한 문단(`hp:p`)을 여러 `hp:t` 조각으로 쪼갠다
   (주입 로직이 파편 텍스트를 지우던 이유). 조각을 순서대로 이어 붙이고 줄바꿈·탭을
   살려 "깨끗한 문단 문자열"을 복원해야 함.
2. **내용 기반 정렬(alignment)**: 판본 간 안정 ID가 없어 문단을 위치가 아니라
   내용 유사도로 짝지어야 함(조항 삽입 시 뒤가 다 밀리므로 단순 라인 diff 부적합).
3. **문단 내 단어 단위 강조**: "요율 3%→3.5%", "1억→2억" 미세 변경을 문장 안에서
   하이라이트(difflib 토큰 diff).
4. **표 대조**: 표를 제목/위치로 매칭 후 셀 단위 비교.
5. **이동 감지**(선택): 이동한 절을 삭제+추가로 오인 안 하기.

**리뷰어 킬러 기능 — 변경 항목 추출**: 검토자가 가장 신경 쓰는 건 바뀐 **숫자**
(기준금액·요율·기한·수치)와 **조항 신설/삭제**. 원시 diff 대신 이걸 우선순위
리스트로: `3.2조 기준금액 1억→2억`, `제5조 신설`, `별표2 3행 단가 변경`.

**사람이 쓰기 좋은 출력**:
- **색상 HTML 리뷰 리포트**: 추가/삭제/수정 색분류 + 문단 내 단어 강조 + 섹션 접기
- **변경 요약**: "문단 12 추가 / 5 삭제 / 8 수정, 표 2개 변경"
- **변경 항목 리스트**(위 킬러 기능)

**재사용/신규**: `HwpxPackage`(컨테이너)·XML 순회는 재사용. 파서에 **본문 텍스트
추출기**(문단·표 단위 정규화)를 신설 — 필드 주입과는 독립된 별개 능력이자
L2 diff·향후 검증·병합의 공용 토대.

설계하는 게 아니라 *따라나온다*.

*동기(재개 시 참조).* 지금 최상위 GUI 표면이 **매핑 위저드(하나의 태스크)**다. 위저드는
홈이 아니라 선형 흐름이고, 앱은 이미 여러 능력을 품는다. 최상위엔 능력들을 **라우팅하는
오케스트레이터(메인 대시보드)**가 와야 한다. 또한 최종 사용자 배포 산출물이 없다
(현재 `pip install -e` 개발 설치뿐). 이 둘을 트랙 C로 묶는다. **단, 위 원칙에 따라
기능 안정화 이후에만.**

*현 표면 vs 능력의 간극(대시보드가 드러내는 것):*
| 능력 | 코어 | GUI 표면 | CLI |
|---|---|---|---|
| 문서 생성(메일머지) | `batch`/`engine` | ✅ 위저드 + `main_window`(구 직결, 중복) | ✅ 기본 |
| 규격서 개정 diff | `core/diff` | ❌ **없음** | ✅ `diff` |
| 템플릿 관리(schema/fieldize/lint/drift) | `core/*` | ❌ **없음** | ✅ 4개 |

→ 대시보드는 단순 런처가 아니라 **비어 있는 두 표면(diff·템플릿 관리 GUI)을 채우는**
작업까지 끌고 온다. 스코프를 명시적으로 인정하고 시작할 것.

- **C-1 대시보드 셸(오케스트레이터).** `gui/app.py` 진입을 `Dashboard`(QMainWindow)로

  방식**(카드 → 기존 위저드/뷰 기동), 상태 통합은 나중. `QWizard`(QDialog)는 임베드가

  상태 소유: 최근 파일/프로젝트, **매핑 프로파일 레지스트리**(열린 질문의 그 자리),
  설정(나라장터 ServiceKey·기본 출력 폴더). 부수 정리: `main_window` 직결 생성은 위저드와
  중복 → "빠른 생성"으로 흡수하거나 제거.
- **C-2 비어 있는 표면 채우기.** ① diff 리뷰 GUI(두 판본 선택 → 색상 HTML 리포트 내장
  뷰 + 변경 항목 리스트), ② 템플릿 관리 패널(schema/fieldize/lint/drift를 dry-run·미리보기
  기본 그대로 GUI로). 코어는 이미 있음 — 표면만.
- **C-3 exe 패키징.** **PyInstaller `--onedir` + repo에 spec 커밋 + `packaging/` 빌드
  스크립트.** 런타임 의존성엔 넣지 않고 dev/build 전용 격리(현 optional-deps 규율 유지).
  유의: PySide6 무거움(~150–200MB) → 미사용 Qt 모듈(WebEngine 등) 제외로 축소; `lxml`은
  바이너리 휠 무난, `openpyxl` 순수 파이썬; `cli.py`의 **지연·동적 import는 hidden-import/hook
  점검** 필요. **한글 COM PDF 경로(열린 질문)는 번들 불가**(호스트 한글 설치 전제) → optional
  모듈 분리 유지. SmartScreen 회피용 **코드 서명**은 장기 과제. 대안 Nuitka(더 작고 빠름,
  상품화 유리)는 후속 저울질 — 지금은 PyInstaller가 마찰 최소.
