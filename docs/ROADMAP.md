## 현재 상태 및 다음 착수 (2026-07-10 갱신)

**기준 스냅샷(2026-07-10 당시): master · pytest 271 통과 · 워크트리 없음(메인만).**

**▶ 나라장터 Open API 현황 정정 (2026-07-12).** 종전의 “웹 취득 + 매핑 UI 완료”는
백엔드/CLI 능력과 제품 GUI 표면을 한 항목으로 합쳐 기록한 과대 표기였다. 실제로는
`NaraStdDataSource`·매핑 코어·CLI 연결과 exe 번들 포함까지만 완료됐고, 앱 B의 위저드와
실행 화면은 여전히 Excel/CSV 파일만 선택한다. 따라서 **나라장터 GUI 소스 선택·개인
ServiceKey 관리·검색 조건·취득 오류 UX는 미완료**이며 아래 B-3b를 다음 제품 기능으로 둔다.
OpenAI/GPT/LLM API는 이 저장소의 설계나 구현 범위였던 적이 없으며, 여기서 “Open API”는
나라장터 공공데이터 API를 뜻한다.

  로직 단일화. 범주 분리: 생성자(`{{date}}`·`{{seq}}`=now/배치상태, `naming` 고유) ·
  결합자(N→1, `mapping` 고유) · 포매터(1→1, 공용). `mapping.apply_transform` 은 시그니처·
  프로파일 스키마 불변(결합 후 공용 포매터에 위임). 미지 포매터는 토큰 통째 유지 + 신고.
- **표시형(format) 선언 — 매핑 위저드의 WYSIWYG + 교체 가능한 서식 엔진**
  (`core/format_engine.py` + `FieldMapping.fmt`(서식 코드) + 매핑 테이블 '표시형' 열).
  **타입 논쟁의 착지점**: 타입은 소유하지 않는다 — 휴리스틱이 *추천*(스키마 배지)하고, 사람이
  매핑 시점에 **변환된 출력을 실 레코드로 보며(WYSIWYG 미리보기+스텝퍼)** 표시형을 고르고,
  프로파일에 **기억**(JSON `fmt`, 하위호환)하고 편집한다.
  - **어댑터 층**(`format_engine.FormatEngine` 프로토콜 + `ENGINE` 교체점): 서식 해석기를
    갈아끼울 수 있다. 지금은 **stdlib**(금액=`str.format` 스펙 `{:,}원`·`{:,.2f}`, 날짜=`strftime`
    `%Y-%m-%d`) — 의존성 0. **날먹 경로**로 표현력이 프리셋에 안 갇힘. babel(CLDR/엑셀 패턴)은
    후일 `ENGINE` 한 줄 교체로 가능(단 우리 축과 강점 안 겹쳐 보류 — 로케일 통화 ₩≠원 등).
  - **fmt = 서식 코드**("" = 기본 원/한글). 프리셋(원·숫자·소수·ISO·점·연월)은 **분류해 클릭하는
    씨앗**일 뿐, 고급 사용자는 '직접 입력…'으로 코드 자유 편집(Excel 갤러리+커스텀). 파싱→서식→
    실패 시 원본(degrade) — 타입 *주장*이 아니라 관대한 *시도*, WYSIWYG가 안전망.
  - 표시형 변경은 확정 해제(사람 눈 재확인).

**▶ 확정 승인된 다음 백로그 (2026-07-09, 사용자 승인).** 표시형/서식 아크의 마무리 3건:
- ✅ **D-6 text_render 프로파일 경유 + 인라인 포매터 제거** — `render` 에 `--profile` 추가:
  소스 레코드에 `profile.apply` 로 **표시형까지 서식된 값** → 순수 `{{field}}` 치환(HWPX 경로와
  동일 모델). 인라인 토큰 포매터(`{{field|amount}}`)는 폐기 — `core/formatters.py` 삭제,
  text_render 는 순수 치환, naming 은 평문 치환으로 복귀. 텍스트·HWPX·파일명이 모두 "프로파일이
  서식한 값을 꽂을 뿐" 으로 통일. e2e 확인(원본 vs --profile).
- ✅ **D-7 마스크 프리셋** — 전화(`010-1234-5678`, 자릿수별 3-4-4/02-4-4/3-3-4)·사업자번호
  (`123-45-67890`). stdlib 서식 스펙으로 안 되는 자릿수 마스크라 `format_engine` 에 작은 함수
  (`_mask_phone`·`_mask_biz`)로 추가하고, **`join`(그대로) 변환의 표시형**으로 노출(전화 필드 =
  변환 그대로 + 표시형 전화). 자릿수 안 맞으면 원본(degrade). 어댑터 이음새 무변경.
- ⏸ **D-8 총동원 타입 추천기 — 파킹(2026-07-09 결정).** 이름 추론만인 현 추천을 **값
  샘플링·소스 메타**까지 확장. **책상 위 추정으로는 한계** — 실데이터의 오추천 사례(예:
  전부-숫자인 계약번호를 amount 로 오추천) 없이 만들면 "그럴듯하지만 틀린 추천"을 낳는다.
  WYSIWYG 확정이 안전망이긴 하나, 추천 품질 자체는 실데이터로만 검증된다. **실사용 데이터
  확보 후 재개.** (표시형/서식 아크는 D-1~D-7 로 완결.)

**개발 방식(반복 적용):** 기능마다 **격리 git 워크트리 + 전용 브랜치**에서 **자율
백그라운드 에이전트**로 개발 → 메인이 **직접 재검증(pytest + 불변식 + 변이 테스트로
회귀 스위트가 하중을 받는지 확인)** 후 **fast-forward 병합**. 파서 원칙: **충실도
완전 · 기능 최소 · 누락은 시끄럽게**. **미완 작업의 워크트리 병렬 분할(파일 풋프린트·의존
웨이브·충돌 핫스팟)은 [WORK_UNITS.md](WORK_UNITS.md)** — 에이전트별 착수 브리프.

**▶ 기능 안정화 단계 완료 (2026-07-09, 사용자 판단). 다음: UX 결정 + UI 구현.**
트랙 B(B-1~B-4)·트랙 D(D-1~D-7, 표시형 아크) 랜딩으로 **생성기·서식·매핑 기능이 CLI에서
안정화**됐다. 이는 트랙 C(제품 셸/UI)를 동결했던 **해동 조건이 충족**됐다는 뜻이다 —
"기능이 안정화되면 UI가 기능에서 자연히 따라나온다"(아래 트랙 C).

**다음 할 일 = UX 결정 + UI 구현(트랙 C 해동).** 단, 관통 원칙 유지: **UI는 기능이 거쳐

서식)·표 행복제는 경계선상 — 위 명시성 원칙으로 판정(명시 선언+미리보기면 허용).

넘어 **실데이터 정답 대비 재현율/정밀도**를 검증하려던 트랙(테스트 피라미드의 빠진
조각). 트랙 B 이후 재개. 필요 파일 3종(law.go.kr, 개정폭 작은 법령): ① 개정 전 HWPX
② 개정 후 HWPX ③ **신구조문대비표**(정답 오라클 — old|new 2열이라 문단 수준 정답에
가장 가까움). 절차: ⓐ 추출기가 법령 구조(조/항/호/목·별표·부칙) 온전 처리하는지

## 1. 데이터 입력 어댑터 층 — UX 보조

**문제**: `DataSource`(`data/base.py`)는 `records()`+`fields()`뿐이라 템플릿
필드명(`진행상태`, `계약명` 등)과 데이터 컬럼 헤더가 **정확히 일치해야만** 채워진다.
현실에선 거의 안 맞는다. 원본 VBA는 이걸 `frmErpPreview`+`modFuzzyMatch`
(Unknown 항목 수동 매핑 HITL)로 이미 풀었다 — 그 아이디어를 되살린다.

**제안: `DataSource`와 `Engine` 사이에 매핑 계층**
```
템플릿필드  ←  Mapping  →  { 소스컬럼 | 상수 | 변환식 }
```
- **퍼지 자동 제안**: 템플릿 필드 vs 데이터 컬럼 유사도 추천(한글 자모 정규화 +
  부분일치). 사용자는 확인/수정만.
- **매핑 프로파일 저장**(템플릿별 JSON): 같은 레이아웃이면 다음부터 자동 적용.
  → **ERP 연동의 결정적 이득**. ERP 필드코드(`CNTRCT_NM`)는 템플릿 한글명과
  안 맞으니, "코드→템플릿명" 매핑을 프로파일로 고정하면 `ErpApiDataSource`로
  갈아끼워도 매핑 재사용.
- **변환/서식 계층**: 날짜·금액 포맷팅(원본 `modDataHydrator`/`modFormatUtils`의
  WYSIWYG 역할). 매핑 항목별 optional transform.
- **데이터 소스 마법사**: 헤더 행 자동 감지, 시트 선택, CSV 인코딩 감지
  (`ExcelDataSource`의 `header_row`/`sheet` 인자를 UI로 노출).

의존: 템플릿 스키마(축)가 필드 목록+타입을 주면 매핑 UI 좌측 열이 자동 생성.

## 3. 파서 고도화로 얻을 향후 기능들

계약·구매 업무 가치 순:

- **① 반복 영역/표 채우기 (최대 가치)**: 품목 리스트처럼 **행 수 가변 표**를
  데이터로 채우기. `hp:tbl`/`hp:tr`/`hp:tc` 이해 + 행 복제. 구매·계약 문서의
  핵심(품목·수량·단가). 사실상 "메일머지 2.0".
- **② 템플릿 스키마 추출/내보내기 (토대)**: `required_fields()`를 확장해
  필드 타입·표 영역·위치까지 JSON 스키마로 출력 → **입력 폼 자동 생성**(1번 매핑
  UI 자동 구축), 검증 강화(미치환 `{{}}` 잔존 검사).
- **③ 리치 값 주입**: 체크박스 on/off, 여러 줄, 서식 있는 런.
- **④ 이미지/직인·로고 삽입**: `BinData` + `manifest.xml` 조작. 복잡하나 공문 가치 큼.
- **⑤ 출력 검증**: 생성물의 빈 필드/미치환 토큰 자동 확인, 부분 실패 정직 보고.

### 3-1. HWPX 규격서 개정 diff — ★ 1급 기능 (핵심 활용처)

**동기**: 규격서·표준서식은 **해마다 개정**된다. 무엇이 어떻게 바뀌었는지 사람이
수작업으로 대조하는 건 느리고 실수 잦다. 두 판본을 넣으면 변경점을 **사람이 훑기
쉽게** 보여주는 도구. → 생성기와 대등한 1급 기능. 독립 "규격서 개정 리뷰 도구"
상품성도 있음.

**성격 재정의**: 이건 채워진 문서의 **필드값 비교(L3)가 아니라 본문·조항·표의 내용
비교**다. 규격서 개정은 누름틀이 아니라 조항 문구·기준 수치·표가 바뀐다. 무게중심:
- **L2 본문 텍스트 diff = 주역**, **표 diff = 필수**(기준표·요율표)
- L1 컨테이너(엔트리 추가/삭제, 해시 비교) = 보조 신호
- L3 필드값 = 부차

**함정**: HWPX XML엔 랜덤 ID가 가득(`id="2073595120"`, `fieldid="627272811"`).
원시 diff는 ID·공백·속성순서 잡음에 파묻힘 → **의미 기반 diff** 필수.

**진짜 어려운 지점 (설계의 본체)**:
1. **문단 텍스트 재구성**: HWPX는 한 문단(`hp:p`)을 여러 `hp:t` 조각으로 쪼갠다
   (주입 로직이 파편 텍스트를 지우던 이유). 조각을 순서대로 이어 붙이고 줄바꿈·탭을
   살려 "깨끗한 문단 문자열"을 복원해야 함.
2. **내용 기반 정렬(alignment)**: 판본 간 안정 ID가 없어 문단을 위치가 아니라
   내용 유사도로 짝지어야 함(조항 삽입 시 뒤가 다 밀리므로 단순 라인 diff 부적합).
3. **문단 내 단어 단위 강조**: "요율 3%→3.5%", "1억→2억" 미세 변경을 문장 안에서
   하이라이트(difflib 토큰 diff).
4. **표 대조**: 표를 제목/위치로 매칭 후 셀 단위 비교.
5. **이동 감지**(선택): 이동한 절을 삭제+추가로 오인 안 하기.

**리뷰어 킬러 기능 — 변경 항목 추출**: 검토자가 가장 신경 쓰는 건 바뀐 **숫자**
(기준금액·요율·기한·수치)와 **조항 신설/삭제**. 원시 diff 대신 이걸 우선순위
리스트로: `3.2조 기준금액 1억→2억`, `제5조 신설`, `별표2 3행 단가 변경`.

**사람이 쓰기 좋은 출력**:
- **색상 HTML 리뷰 리포트**: 추가/삭제/수정 색분류 + 문단 내 단어 강조 + 섹션 접기
- **변경 요약**: "문단 12 추가 / 5 삭제 / 8 수정, 표 2개 변경"
- **변경 항목 리스트**(위 킬러 기능)

**재사용/신규**: `HwpxPackage`(컨테이너)·XML 순회는 재사용. 파서에 **본문 텍스트
추출기**(문단·표 단위 정규화)를 신설 — 필드 주입과는 독립된 별개 능력이자
L2 diff·향후 검증·병합의 공용 토대.

  - **홈 = 작업 목록(레지스트리) + 「새 작업 만들기」.** (미결이던 매핑 프로파일 레지스트리 =
    이 목록으로 승격.)
  - **위저드 = 홈이 아니라 "작업 에디터"로 강등**(템플릿→샘플데이터→매핑 확립→**작업 저장**;
    마지막이 "생성"이 아니라 "저장").
  - **생성 = 별도 "실행(Run)" 화면**(작업 고름 → 데이터·행 겨눔 → 미리보기 → 생성). 매핑
    재확정 없음 — 무거운 명시성 게이트는 셋업(에디터)으로만.
  - **`main_window` 삭제**(위저드의 매핑-뺀 부분집합 = 명시성 게이트 위반, "빠른 생성" 발명의 씨앗).

- ✅ **여러 데이터 → 한 양식 = 접근 문제, 조인 파킹 (목표 형태 확정 — ADR K).** 원본 조인규칙 =
  **나라장터/내부DB의 디지털 트윈 우회로**(직접 접근 부재의 산물). 커널은 "조인 필요"가 아니라
  "**한 양식엔 완성된 엔티티 레코드 필요**" — 서버가 조립해 주면(직접 API/DB) 조인 불필요. 그
  접근이 **지금 없음**(나라 표준=목록까지, 세부 op 미착; ERP API 미확정). → **지금 조인 안 만듦**.
  조립은 **`DataSource` 이음새 뒤**에 삶(생성·작업·실행과 직교) → 작업의 데이터 측을 `DataSource`로
  **추상 참조**, 단일소스 가정 못 박지 않음. **갱신(2026-07-12): 파킹 근거를 "철학"에서 "수요"로
  이동, 목표 형태 확정.** `UI_DESIGN_DECISIONS.md` **ADR K**: 조립은 착지 시 **사용자 저작·
  미리보기의 Power-Query식 파이프라인**(외부 엔진 래핑, `DataSource` 생산자, 추론 엔진 아님)으로
  landing — 명시 저작이 "책상 위 추정 금지"를 해소하므로 남은 파킹 근거는 **수요**(실제 2+소스
  워크플로)뿐. 파이프라인 = **저장되는 데이터 *가공방식* 레시피**(데이터 아님, 참조만; 한 겹
  불변식 유지). merge 키 = 구조축(schema)·명시·미리보기(UnivContractor #2). ERP API·나라 세부
  op와 함께 착지하는 **데이터-접근 파킹 항목**.

- ✅ **(C) 컴파일 상태 가독성 — 저작 측 coverage ledger (2026-07-12 토의).** (B)의 **거울상**:
  (B)=출력 coverage(빈칸을 시끄럽게), 이건 **입력 coverage**(미컴파일을 시끄럽게). 둘 다 파서
  coverage ledger의 제품화. **파서를 완벽히 만들어도(skip 0) 남는 직교 문제** — 완벽한 파서도
  RAW/COMPILED/FILLED 구분은 못 만듦.
  - **문제.** `.hwpx` 하나가 **RAW**(필드0+`{{}}`토큰) / **PARTIAL**(필드有+skip·stray 잔존) /
    **COMPILED**(필드有+stray0+값=`{{X}}` 리터럴) / **FILLED**(값≠placeholder) 중 어디인지 파일만
    봐선 구분 불가. CLI `fieldize`는 fire-and-forget이라 **상태를 소유·질의하는 주체가 없음** →
    사람이 템플릿 라이브러리 관리 시 "완전한 문서 vs 컴파일 전 확인 필요 문서" 구분 실패.
    **변형(transform)은 상태를 소유 못 함 → 관리 표면 필요**(= C-2의 구조적 정당화, 편의 아님).
  - **원칙: 저장 말고 계산.** 파일에 "컴파일됨" 도장 금지 — 한글 재편집에 드리프트(거짓말). diff의
    "랜덤ID 불신"·txt의 "view=진실"과 같은 결. `scan_tokens`+`extract_schema`에서 온디맨드 파생.
  - **표면 3겹(배지 + 게이트 + 인라인수정, 한 계산속성 `compile_status`의 렌더러).** 템플릿을 만나는
    세 순간에: **홈 앰비언트 배지**(기존 `template_missing` pill 어휘 확장; `refresh`마다 재계산 —
    한글 편집으로 COMPILED→PARTIAL 드리프트하므로) / **위저드 게이트**(RAW 이미 차단; **PARTIAL을
    비차단 경고→확정 게이트로 승격** — [여기서 컴파일] 또는 명시적 ack, ADR-E 의도적공란 `_acked`
    패턴 재사용) / **관리 패널 워크숍**(상태별 안전연산 게이팅: RAW→컴파일, PARTIAL→마저 컴파일,
    COMPILED→사용).
  - **지렛대 = PARTIAL을 조용히 통과 불가로.** raw는 이미 차단, compiled·filled는 안전. 새 설계
    무게는 전부 "성공처럼 보이나 skip 토큰을 fill이 조용히 흘리는" PARTIAL에. confirm-or-alarm 그대로.
  - **결측 프리미티브:** COMPILED vs FILLED 구분엔 **필드 값 읽기 API** 필요(`run_state.py:88`에서
    이미 파킹). 그전까진 GUI는 **3-상태(RAW/PARTIAL/COMPLETE)만 정직 표시**, FILLED 인스턴스는 관리
    스코프 밖(추측 안 함). → **필드 값 읽기 = 4-상태 완성의 코어 선결 과제**.
  - **금지:** 파일명 관례(`_compiled.hwpx` = 암묵 상태를 파일명으로 이전, 같은 병) · 파일 내 도장
    (드리프트) · 순수 장식 배지(액션 게이팅 없으면 소프트경고 놓침 반복).
  - **착수 순서:** 코어 `compile_status` 파생 + 계약 테스트(seam 먼저) → 위저드 PARTIAL 게이트
    (가장 값싼 위험 제거) → 홈 배지.

- ✅ **(B) 빈칸 허용 + 시끄러운 표식.** 파서 원칙("누락은 시끄럽게")을 **출력으로** 확장(파서
  coverage ledger의 생성기 짝). 실행 시점 **능동 게이트**: "빈칸 N필드 — 허용하고 생성?"(필드
  나열; `validate.empty_valued`를 수동 로그→게이트로 승격). **미충족 공란**(있어야 할 데 없는)만
  이름붙은 **grep 가능 표식**(예 `〘미입력·{필드}〙`) 주입, **의도적 공란**(매핑 `비움`)은 표식
  없음(경보 피로 방지). 표식은 **누름틀 유지**(한글에서 클릭 채움 + 눈에 보임 둘 다). 로드맵
  ⑤ 출력검증 = 표식 grep. 원본 pre-flight(헤더 없으면 중단)를 **넘어섬**(부분 문서를 내되
  외치게 — 트리클-인에 적합).

- ✅ **누적 치환(cumulative fill) 허용.** 여러 데이터→한 양식의 **시간축 해법**: 한 순간 조인
  대신 **행정기간 단계마다 그 단계 레코드로 문서를 누적**. **기술적으로 이미 가능(엔진
  무변경)** — 확인함: `set_field`가 누름틀 컨트롤 보존(텍스트만 교체, `fields.py:105`),
  `required_fields`가 채운 뒤도 재발견(`fields.py:48`), `engine.py:42`가 **빈값 스킵**(이전
  패스 값 안 덮음), 출력이 OCF 온전 HWPX라 **round-trip 템플릿**. 빠진 건 능력이 아니라
  워크플로("이전 출력을 이어채운다"). **각 단계=작업**, **(B) 표식=다음 패스 슬롯**과 한 몸.
  규칙: 단계별 필드 **서로소**(겹치면 덮어씀), **문서가 상태 원장**(외부 대장 불필요),
  **단건 자명·배치 누적은 "이전 출력↔레코드" 파일키 매칭**(데이터 조인 아님). → **실행의
  "데이터 겨눔"이 두 대상**: 새 템플릿(신규) | 이전 출력(누적).

- ⏸ **스톱갭 (A) 엑셀 임시 배관.** 손 조립 시트 → `ExcelDataSource` 그대로, **새 코드 0**.
  (B)와 보완(손으로 이은 만큼 채우고 잔여는 표식).
