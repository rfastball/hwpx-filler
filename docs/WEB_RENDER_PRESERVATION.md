# 웹 재렌더 시 상호작용 보존 — 전체 스냅샷+복원 vs 부분 패치 (#28)

웹 프론트엔드가 Python→웹 푸시로 화면을 다시 그릴 때 사용자의 **포커스·캐럿·스크롤**을
어떻게 지킬지에 대한 설계 결정과 그 근거·재고경계를 남긴다. 구현은 `web/js/preserve.js`
(`window.Preserve.around`)와 각 화면 `render()` 래핑.

## 맥락 — 왜 유실이 생기나

현 아키텍처는 의도된 단순 모델이다(마이그레이션 북극성):

> **Python 이 상태를 단일 소유 → 전체 스냅샷을 웹에 푸시 → JS 가 `innerHTML` 로 서브트리를
> 통째 재구성.** "멍청한 뷰"라 에이전트가 저작하기 쉽고, 진실 원천이 한 곳(Python)이다.

대가는 재구성이 노드를 파괴한다는 것: 사용자가 프리뷰를 스크롤해 뒀거나 입력칸에 커서를
둔 채 푸시가 오면, 재구성 후 스크롤은 맨 위로 튀고 포커스·캐럿은 사라진다. 조용한 상태
유실은 이 앱의 원칙(**confirm-or-alarm** — "묻고 확정하게 하라, 아니면 시끄럽게")에 반한다.

## 결정

**전체 스냅샷 모델을 유지하고, 렌더 직전 캡처→직후 복원하는 얇은 옵트인 헬퍼를 둔다.**
부분 패치/vdom 으로 전환하지 않는다.

- `Preserve.around(renderFn)`: 렌더 직전 포커스 요소(id)·캐럿/선택 범위·`[data-preserve-scroll]`
  컨테이너의 스크롤을 캡처하고, `renderFn()`(기존 `innerHTML` 재구성) 실행 후 같은 id 로
  복원한다. 복원 대상이 재구성을 가로질러 사라졌으면 no-op(= '보존할 것이 없음'이라는 정상 귀결).
- 스크롤은 **옵트인**(`data-preserve-scroll`): 진행 로그(`genLog`·`mxGenLog`)처럼 **바닥 고정**
  의도가 있는 컨테이너를 무차별 보존이 덮어쓰지 않게. 진행 델타 푸시는 `render()` 초입에서
  early-return 하여 애초에 `Preserve` 밖에 둔다.
- **범위 경계**: 헬퍼는 포커스·캐럿·스크롤 **연속성**만 책임진다. 결과 수명주기 무효화(오래된
  생성 결과를 새 스냅샷이 지우는 것)는 컨트롤러(Python) 소유로 이 헬퍼 밖이다.

### 현재 실효 범위 — 스크롤은 활성 보장, 포커스·캐럿은 잠재(latent)

정직하게 밝힌다: 복원은 **재구성을 가로질러 같은 id 를 유지하는 요소**에만 걸린다.

- **스크롤 = 활성 보장.** 옵트인 컨테이너(`renderView`·`tokPanel`·`editor-body`·`recList`·
  `mxJobList`·`mxRecList`)는 `index.html` 의 정적 요소라 안정 id 를 갖는다 → 실제로 보존된다.
- **포커스·캐럿 = 잠재.** 현 4화면에서 `render()` 가 재생성하는 편집 컨트롤은 대부분 **안정 id 가
  없다**(에디터 폼은 `readonly` 필드, 매핑 표는 이벤트 위임 `select`, 레코드 리스트는 체크박스 —
  모두 캐럿을 갖지 않거나 id 가 없다). 즉 재렌더되는 자유 입력 텍스트 필드가 **현재는 없어서**
  포커스·캐럿 복원이 실효를 낼 대상이 없다. 헬퍼의 로직은 정확하며, 어떤 컨트롤이든 안정 id 를
  얻는 순간 자동으로 활성화된다 — 지금은 대기 상태다.

이 잠재성은 위 절차성 논지와 정합적이다: 이 앱은 애초에 push-render 경로에 라이브 편집 텍스트
필드를 두지 않는다. 그래서 포커스·캐럿 보존은 "미래를 위해 켜 둔 정확한 기제"이고, 오늘의
실질 가치는 스크롤 연속성이다.

## 근거 — 왜 패치가 아니라 복원인가

1. **푸시가 coarse 다.** 이벤트 배선을 보면 푸시는 `change`·`click`·스텝 등 **이산 액션**에서만
   발생하고 `input`(글자 단위)에는 물려 있지 않다. 즉 **타이핑 도중 재구성이 없다** → 복원이
   레이스 없이 충분하다. (한글 IME 조합 중 재구성이 끼어 음절이 깨질 일도 구조적으로 없다.)

2. **절차성이 라이브 편집을 배척한다.** 라이브 타이핑-인터랙티브는 *연속·무게이트·즉시 반영*
   인데, 이 앱의 척추는 정반대(*조립 → 확정 → 작용*, 마법사·강제 확인 게이트)다. 증거로,
   템플릿 편집을 상시 인페이지 라이브 필드가 아니라 **모달**(저장 시 확정되는 이산 절차 단위)에
   넣었다. 부분 패치의 이득(키스트로크 단위 포커스 유지)은 이 앱에서 **구조적으로 오지 않는다.**

3. **북극성 정렬.** 복원 헬퍼는 순수 DOM 부기(bookkeeping)라 "멍청한 뷰·Python 단일소유"를
   지킨다. 패치/vdom 은 상태 재조정 로직을 뷰로 들여 진실 원천을 쪼개고, 시간이 갈수록
   에이전트 저작성(마이그레이션의 동기)을 갉는다.

4. **엑싯이 싸다 → 미루기에 리스크가 없다.** `Preserve.around` 는 순수 봉투라 패치로 갈 때
   **삭제**하면 되고(마이그레이션 아님), 부분 패치의 실비용도 morphdom 류(무의존·CSP상 인라인)
   로 `host.innerHTML = html` → `morph(host, html)` **커밋 한 줄 교체**라 낮다. 게이트 단언
   ("푸시 후 포커스·스크롤 생존")은 패치에서도 참이라 **그대로 전이**된다. 락인 0.

정리하면 패치의 이득은 구조적으로 안 오는데 비용(북극성 침식·유지 부담)은 즉시·상시라, 실제
강제함수가 나타날 때까지 미루는 게 옳다(YAGNI).

## 재고경계 — 언제 패치를 다시 볼까

단순 성장이 아니라 **근본 피벗**일 때만:

> 앱이 **confirm-or-alarm 절차성을 버리고 라이브 편집면을 도입**할 때 — 특히 **한글 텍스트
> 필드에 글자 단위 푸시**가 붙을 때.

해로운 케이스는 아래 **세 조건이 동시에** 성립해야 한다:

1. 푸시가 `input`/`keyup`(글자 단위)에서 발생 — `change`/`blur`/`click` 아님
2. 포커스된 편집 노드가 `render()` 재구성 서브트리 **안**에 있음
3. 재구성이 full `innerHTML`(노드 파괴)

하나라도 거짓이면 복원으로 충분하다. 가장 뾰족한 트리거는 한글 IME: 조합(가→각→간…) 도중
재구성이 끼면 진행 음절이 커밋/유실되는 **경성 실패**가 난다.

**진단 증상**: 캐럿이 끝/처음으로 튐 · 글자 드롭 · 프리뷰 깜빡임 · 입력 랙 · IME 조합 글자
깨짐/중복.

**에스컬레이션 순서**: (a) 해당 라이브 필드 하나만 재구성 서브트리에서 빼는 **표적 제외** →
(b) 그런 면이 여럿으로 번지면 morphdom 커밋 교체(부분 패치). 전면 vdom 재작성은 마지막.

## 검증

- 정적(`tests/test_web_dom_contract.py`): `preserve.js` 로드 + 3개 화면(`txt`·`editor`·`run`)
  `render()` 의 `Preserve.around` 래핑 존재 — 헤드리스 포함 전 플랫폼.
- 동적(`tests/test_web_selftest_gate.py` + `--selftest` 프로브):
  - **기제**: 합성 픽스처로 `innerHTML` 재구성을 가로질러 포커스·캐럿(선택 범위)·옵트인 스크롤이
    실제로 보존되는지 실 WebView2 에서 되읽어 단언.
  - **실화면 회귀**: shipped `window.__push` 경로로 3화면(`txt`·`editor`·`run`) 실
    `render()` 를 실 컨트롤러 스냅샷으로 구동해 `Preserve.around` 래핑이 render 를 깨지 않는지,
    그리고 `txt` 프리뷰의 스크롤이 실 재렌더를 가로질러 유지되는지(강제 오버플로) end-to-end 단언.
