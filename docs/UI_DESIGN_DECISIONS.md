# UI 디자인 표면 결정 기록 (ADR)

이 문서는 **앱 B(생성기)의 UI 표면 결정**을 닫은 근거·트레이드오프·출처를 남긴다.
[UI_CONTRACT.md](UI_CONTRACT.md)가 *무엇을*(ViewModel seam)의 계약이라면, 이 문서는
*왜 그 형태인지*의 결정 기록이다. 관통 원칙 —
**function → UX → UI, 그 역이 아니다.**

## 방법

세 겹의 조사로 각 결정을 닫았다. 앞의 둘은 계보-기술(descriptive), 셋째는 결정-지향.

1. **함정 계보 조사** — 문서자동화 3계보(법률 문서조립·대중 메일머지·데이터→문서 생성기)가
   반복해서 밟은 UX 지뢰. 핵심 발견: **조용한 실패(silent field mismatch)가 카테고리의 원죄**.
2. **UI 형태학 조사** — 그 UX 압력이 굳어진 UI 수렴/분기. 핵심 수렴: **델리미터 토큰·삽입
   팔레트·필드 매핑 대화상자**. 핵심 분기: **"라이브 분할 미리보기는 표준이 아니다".**
3. **결정-지향 조사** — 단건·고위험·저빈도·비개발자 데스크톱 채움이라는 우리 프로파일에
   각 선택지를 겨눈 증거(미리보기·빈칸 표면화·Home 진입).

각 조사는 팬아웃 검색 → 소스 페치 → 3표 적대 검증(2/3 반증 시 기각) → 종합으로 수행됐다.

## 트랙 이원성 — HWPX(무거움) · txt(가벼움)

앱은 **두 출력 트랙**을 가진다. 아래 결정 중 일부는 트랙별로 갈라지고 일부는 공유된다.

| 축 | HWPX 트랙 | txt 트랙 |
|----|----------|---------|
| 입력 | `.hwpx` 누름틀 템플릿 | `.txt` `{{필드}}` 템플릿 |
| 처리 | OCF/lxml (무거움) | 순수 문자열 치환 (가벼움, `text_extract`의 거울) |
| 산출 | 생성된 `.hwpx` 파일 | 실시간 렌더 텍스트 view + 클립보드 복사 |
| 용도 | 정식 문서 생성(Job-앵커) | 즉각 붙여넣기(온나라 기안 등) |
| 검증 표면 | 한글(외부 권위 렌더러) | **렌더된 view 자체가 진실** |
| 현재 표면 | GUI(home/editor/run) | GUI(홈 기안 카드 → `gui/txt_view`·`txt_state`, ADR H 착지) + CLI(`render --clip`) |

**공유 엔진.** 매핑 + 표시형 서식(`profile.apply`)은 두 트랙 공통 — txt 트랙도 표시형까지 적용된
값을 받는다(`core/text_render.py`는 서식하지 않고 값만 꽂음). 따라서 **엔진을 건드리는
결정(B·D·F·서식)은 트랙 무관 통일**, **출력 표면을 건드리는 결정(C·E의 형태)은 트랙별로
갈라진다.**

## B — 채움/미채움 필드의 시각 구별

**결정.** 필드는 모든 표면에서 본문과 시각적으로 구별된다. 미충족 빈칸은 어디서도 "그냥 빈
공간"으로 보여선 안 된다. 출력물에선 현행 `〘미입력·{field}〙` 누름틀 표식(`core/job.py`
`MISSING_MARKER`)을 유지하고, 앱 내 필드 목록에선 `채움 / 빈칸 / 미입력` 3상태를 각기 다른
배지로 표시한다.

**근거.** 4계보(Word `«»`, HWP `{{}}`, Gavel `{%%}`, docassemble `{{}}`) 전부 플레이스홀더를
델리미터 토큰으로 렌더하며 원시 변수명·빈칸이 본문에 섞이는 형태는 없다. 메일머지 원죄가
"빈 필드 → 경고 없는 빈 줄"이었음을 상기하면, 미충족을 눈에 보이는 토큰으로 승격하는 것이
이 카테고리가 도달한 방어선. 현행 `MISSING_MARKER`는 이미 이 수렴 패턴의 정확한 적용.

**트레이드오프.** 출력물에 표식이 남아 "깨끗한 문서"가 아님 — 원칙상 의도된 소음.

## D — 매핑 UI: 자동매칭 후 미매칭만 수동확정

**결정.** 이름 기반 자동매칭을 채택하되 **"제안(suggested)"으로만** 채운다. 고신뢰(정확 일치)
매칭은 "일괄 수락"으로 확정 부담을 낮추고, **모호·미매칭 필드만 시끄럽게 hard-stop**으로
개별 확정을 강제한다. 자동으로 *조용히 바인딩*하는 경로는 없다.

**근거.** Word `Match Fields`와 HWP `필드 이름 연결`이 독립적으로 "자동매칭 → 미매칭만 수동
확정"에 수렴. 우리 게이트(`MappingModel.is_complete` 전건 확정)와 충돌하지 않는다 — 계보
패턴은 자동 *제안*이지 자동 *확정*이 아니므로, "제안됐으나 미확정" 상태 + 명시적 수락은
`confirm-or-alarm`의 "묻고 확정하게 하라"와 일치.

**트레이드오프.** 고신뢰/모호 2등급 분류가 필요해 순수 전건 클릭보다 구현이 큼.

## F — 소스 열 select-and-insert

**결정.** 소스 열은 실제 데이터소스의 열 목록에서 선택(드롭다운/콤보박스)한다. 필드명 자유
타이핑 경로를 UI에서 제거한다. **엄격 1:1**: 한 누름틀(템플릿 필드)은 정확히 한 데이터 항목을
**단일선택**하고 한 값 유형(**텍스트 / 날짜 / 금액 / 고정값**)으로 서식한다 — N→1 결합·구분자
(sep)·다중소스는 모델에서 제거됐다(코어 미러 `core/mapping.py`). `고정값`(옛 ‘상수’)은 소스 무관
리터럴이고, 날짜 유형 표시형 프리셋은 **시각·날짜+시각**을 포함해 예전 두 키 합치기를 대체한다.

**근거.** 4계보 전부 필드 참조를 자유 타이핑이 아닌 팔레트 선택으로 주입. 결정적으로 이건
1차 조사의 원죄(오타·불일치 → 조용한 미연결; HWP `이름≠성명`)를 UI 층에서 원천 봉쇄한다 —
존재하는 열만 고를 수 있으면 불일치 바인딩 자체가 불가능.

**트레이드오프.** 데이터소스 연결 후에야 열 목록이 뜸 — 현행 흐름(소스 겨눔 → 매핑)과 정합.

## C — 미리보기 모델

**결정.** 인앱 라이브 미리보기를 두지 않는다. 생성된 `.hwpx`를 한글에서 여는 것이
미리보기·검증 표면이다. 전제: (i) 생성이 비파괴적(원본 템플릿 불변, 출력은 새 산출물),
(ii) "생성 → 한글 확인 → 재생성" 루프가 값싸고 즉각적일 것.

**근거.** 미리보기는 비가역 작업 직전 평가의 간극(Gulf of Evaluation)을 메우는 오류예방
표면으로 정당화되나 이는 **원칙(NN/g)이지 벤치마크된 오류감소가 아니다**. 미리보기가 실제로
오류를 줄인다는 직접 실험 증거는 없고(Knauff & Nejasmic 2014는 시각피드백을 변수로 격리 못
함; 수식 과제에선 미리보기 없는 LaTeX가 우세) → "미리보기 없이 원본앱에서 열기"를 무효화할
강한 실증근거도 없다. **결정적 논리(우리 맥락 고유): 한글이 권위 있는 렌더러다.** 인앱에
HWPX 렌더러를 만들면 "우리 미리보기 ≠ 한글 실제 출력"이라는 WYSIWYG 괴리 위험을 자초한다.
출력이 비파괴적 새 파일이므로 재생성 루프가 이미 평가의 간극을 메운다.

**트레이드오프.** 루프에 한글 실행 지연 — "출력 열기"를 즉각·명시적으로 유지해 완화.
인앱 라이브 렌더는 짓지 않는다(이득 미검증 < 충실도 위험 확실).

**트랙 분기.** 위는 **HWPX 트랙에 한정**된다. **txt 트랙은 정반대다** — 평문엔 외부 권위
렌더러가 없으므로 렌더된 view가 곧 진실이자 산출물이다. 여기선 실시간 인앱 view가 충실도
위험이 아니라 트랙의 본질이고, 클립보드 복사가 commit이다. 같은 원리("권위 렌더러를 흉내 내지
않는다")가 HWPX에선 "인앱 렌더 금지", txt에선 "실시간 view가 정답"이라는 반대 표면을 낳는다.

## E — 누락/빈칸 표면화 (차단 모달 → 상시 인라인) · **현행 변경**

**결정.** 미충족 빈칸은 필드 옆 상시 인라인 검증 패널/배지로 표면화한다(온블러 시점 갱신,
절대 빈 공간으로 두지 않음). 생성 직전의 범용 차단 모달("빈칸 N필드… 넣고 생성?" → 예/아니오,
현행 `run_view` 능동 빈칸 게이트)을 주 안전장치에서 내린다. 최종 게이트를 남기려면 범용 확인이
아니라 (a) 미충족 필드를 이름으로 재진술하고 (b) 그 항목을 직접 건드리게 하는 강제
상호작용이어야 한다.

**근거 (이번 조사 중 가장 강함 — 1차 다수 수렴).**
- 반복 차단 모달은 **두 번째 노출부터** 습관화로 무력화(CHI 2017 fMRI+시선; Frontiers 2020:
  반복마다 선형 감소). 범용 확인은 반사적 클릭·경고 실명으로 전락(NN/g).
- **시각적으로 요란하게 만드는 것만으론 못 막는다** — ANSI 고대비도 유의차 없음. **오직 임계
  필드와 강제 상호작용(드래그/재입력)만 습관화에 저항**(SOUPS 2014 Bravo-Lillo).
- 차단 모달은 진짜 비가역·파괴적 작업에만 정당(NN/g "If Not Overused"). 오류 메시지는 필드
  옆 지속 배치가 작업기억 부하 최소(NN/g). 인라인(온블러) 검증이 종단 검증을 이김(Baymard).

**`confirm-or-alarm` 원칙과의 관계 — 위배가 아니라 강화.** "시끄럽게 알려라"의 최적 구현은
습관화로 조용해지는 종단 모달이 아니라 없앨 수 없는 상시 인라인 알람이다. 표식 주입이라는
*결과*의 확정이 필요하면, 그 게이트는 미충족 항목을 직접 건드리게 해 반사적 dismiss를 봉쇄.

**정직한 단서.** 습관화 증거는 대부분 보안 경고 맥락·짧은 간격 반복 가정. 우리는 저빈도라
습관화가 다소 완화될 수 있으나 방향은 견고. "강제 상호작용(SOUPS)"과 "상시 인라인(NN/g·
Baymard)"의 교차점엔 직접 실험이 없음 — 원칙 추론.

**트레이드오프.** 상시 패널은 `QMessageBox` 하나보다 구현이 큼. 그러나 모달은 증거상 스스로
무용해지므로 값싼 게 아니라 거짓 절약.

**트랙 분기.** txt 트랙(`core/text_render.py`)은 이 결정의 **레퍼런스 구현**이다 — 누락 필드는
`{{토큰}}`을 눈에 보이게 남기고(`RenderReport.missing_fields`) 시끄럽게 신고하며 조용한 빈칸
처리를 하지 않는다. HWPX 트랙은 상시 인라인 패널 + 표식으로 이 txt 트랙과 **동등한 표면에
도달**하는 것이 목표다.

## A — Home 진입 모델 (Job-앵커 유지)

**결정.** Job 레지스트리를 기본 진입점으로 유지한다(현행 `home.JobListHome`). 단 (i) 최초/빈
상태는 "첫 작업 만들기" 단일 CTA로 온보딩하고, (ii) 각 job 실행은 저빈도·고위험이므로
마법사식 가이드 흐름을 유지한다(현행 `run_view` 사전검증 흐름).

**근거 — 도메인 사실이 방향성 조사를 확정.** 결정-지향 조사는 *저반복* 프로파일이면
"job-목록 기본 진입은 미스매치, 문서-우선+마법사 직행이 최적"이라고 기울었다(NN/g wizards,
PatternFly, CHI 2007 재발견 종단; 재인>회상은 recents 가시화만 지지). **그러나 이 결론은 저반복
전제에 걸려 있었고, 실제로는 사용자가 같은 서식(템플릿+매핑)을 매우 많이 재사용한다.** 재사용이
높으면 재인>회상이 실제로 적용되고 job 레지스트리는 진짜 재사용 자산 → 기본 진입으로 정당.
개별 *실행*은 저빈도·고위험이므로 그 안의 흐름만 가이드형으로 유지하면 마법사 문헌과도 정합
(진입=자산 목록, 실행=가이드). 이는 핸드오프의 Job-앵커 계약을 독립적으로 검증한다.

**트레이드오프.** 재사용 빈도가 낮아지면 이 결정은 뒤집힌다(문서-우선으로). 재사용률이 결정의
경첩이므로, 실사용에서 재사용이 실제로 유지되는지 관찰 대상.

**트랙 분기.** 위 Job-앵커는 **HWPX 트랙**이다. **txt 트랙은 경량·즉시**(render→copy)라 저장
Job보다 빠른 직행 액션에 가깝다 — A의 방향성 조사가 (저반복 전제에서) 기울었던 "직행/즉시"가
txt 트랙에 자연스럽게 들어맞는다. txt에 GUI를 붙일 때의 진입은 Job 레지스트리와 별도로 설계.

**개정(ADR J).** A의 "진입 = 자산 목록"은 유지하되, ADR J(축 1 착지)가 durable 자산을 **3종으로
확장**한다 — 잡(T+M) · 데이터셋(참조) · 매핑 베이스. 조합은 **한 겹**(일회 실행·저장 안 함).
재사용-앵커는 흐려지지 않는다: 셋 다 재사용 자산이고 진입은 여전히 자산 목록이다.

## G — 신규/누적 대상 문서 선택 ("이어채우기")

**UI 피드백 개정(2026-07-14, #18).** 일반 실행 흐름은 문서를 한 번에 완성하는 **신규 생성만
노출**한다. 누적 이어채우기 구현과 VM seam은 하위호환·별도 고급 진입을 위해 존치하지만 일반
실행 화면에서는 숨긴다. 여러 데이터 결합 요구는 이어채우기로 반복 채우지 않고 작업 구성 전
데이터 관리의 고급 파이프라인에서 하나의 소스로 조립한다. 작업 등록 중 선택한 샘플 소스 참조는
등록 데이터로 자동 등록하되 행 스냅샷·ServiceKey는 저장하지 않는다.

**종전 결정(#18 이전, 보존된 VM seam의 근거).** 대상 문서 선택은 상호배타 2모드 radio를 유지한다: 신규("새 문서 생성", 기본) /
누적("이전 출력 이어채우기"). 선택 지점에 누적 모드의 멘탈모델을 한 줄로 설명한다(누적 =
이전 출력물에 이번 단계 필드를 덧채움 — 시간축 다단 채움). **겹침 정책**: 이전 패스와 이번
패스의 매핑이 공유하는 필드명 수준 겹침은 계산 가능하므로, 현행 수동적 안내(`_sync_prev_note`
비차단 노트)를 **시끄러운 확인 요구 신호로 승격**한다 — 덮어쓰기가 조용히 일어나지 않게.
**범위**: 누적은 단건(1 record) 유지, batch-cumulative는 실데이터까지 parked.

**근거.** 계보에 직접 대응이 없는 순수 설계(시간축은 우리 고유)이므로 이미 닫힌 두 앵커에서
도출한다. ① 관통 원리 #1(불일치·누락을 시끄럽게): 누적 모드의 필드 겹침 덮어쓰기는 잠재적
조용한 실패이므로 loud 확인으로 승격해야 한다. ② 결정 E의 원리(상시 표면화·반사적 dismiss
봉쇄)를 겹침 신호에 준용. radio는 상호배타 모드에 대한 정직한 표현이라 유지.

**정직한 단서 / 격차.** 값 수준 겹침 감지(대상 필드가 *이미* 채워졌는지)는 종전 field-value
read API 부재로 파킹됐다(`run_view` 파킹). 전제였던 API는 C1 `read_field`/`read_fields`
(`core/fields.py`, 커밋 `b76e247`)로 확보됨 — 단 **격상 구현 자체는 미착**이라 현 단계 loud
신호는 여전히 필드명 수준까지만 정확하다(`run_state.set_prev_output` 교집합 고지).
batch-cumulative(이전출력↔레코드 file-key 매칭)는 실데이터 확보 시 착수.

**트레이드오프.** 필드명 겹침을 loud로 올리면 서로소 설계를 이미 한 사용자에겐 마찰. 그러나
조용한 덮어쓰기 위험 > 마찰 비용(원칙상).

## J — 템플릿·데이터·매핑 3풀 분리 + 다대다 실행 · **착지(2026-07-12·J1·J2·J3 병합)**

**상태.** **착지** — **축 1·2·3 및 매핑 모델 모두 2026-07-12 토의로 닫힘**(아래 "축 확정" 블록),
구현 = J1 데이터셋 풀(`89e9cd7`)·J2 매트릭스 실행(`7bbffd1`)·J3 공유 베이스 매핑+오버레이·어휘
워크벤치(`cac94c7`) 전부 master 병합. 생성 전/후 프로버넌스·드리프트는 **ADR L(생성 원장)로
분리·착지**. **조인/조립(여러
소스 → 한 `DataSource`)은 ADR K로 분리·**착지**(2026-07-12, KA·KB 병합). **ADR A(Job-앵커 진입)를 개정**하고 미결의
"H 후속"(txt 인라인 입력)·"A 경첩 관찰"과 로드맵의 파킹 질문 "작업 경계 — 한 겹 vs 두 겹"을 한
실로 흡수한다.

**문제(피드백).** 저작 마법사(`gui/wizard.py`: 템플릿→데이터→매핑→저장)가 **템플릿↔데이터를
항상 1:1로 강제**한다. 사용자 제안: 템플릿과 데이터를 **따로·복수로 등록·싱크**하고, 매핑을
**따로(가능하면 일괄)** 선언. 같은 뿌리의 증상이 txt 트랙에도 있다 — 즉시 기안이 값 몇 개를
넣자고 스프레드시트 파일을 강제한다(`txt_view._pick_data`, H 후속).

**핵심 관찰 — 1:1은 UI 표면 인공물이지 코어 제약이 아니다.** `Job` 은 durable `{템플릿·매핑·
파일명}`이고 **데이터·행을 담지 않는다**(`core/job.py`); 실행은 `RunRequest` 가 그때그때 소스를
겨눈다. 즉 "템플릿과 데이터의 분리"는 **데이터모델에선 이미 참**이다. 1:1로 *느껴지는* 원인은
오직 (i) 선형 마법사가 매핑 저작을 데이터 파일에 결박하고, (ii) 실행 화면이 잡 1개 × 데이터
1개만 다루는 **표면 형태**일 뿐이다. 따라서 이 ADR은 새 엔진이 아니라 **다대다 오케스트레이션
표면의 추가**다.

**결정(제안→착지) — 세 갈래.**
1. **데이터 풀 분리.** 매핑 저작을 데이터 파일에서 떼어 **스키마(템플릿 필드)만으로** 확정한다
   (샘플 데이터는 선택적 미리보기로 강등, 필수 아님). 데이터는 실행 시점에 풀에서 겨눈다.
   이때 **H 후속(파일 없는 수동 인라인 입력)은 자연히 흡수**된다 — 손으로 친 1건 레코드는
   `DataSource` 이음새 뒤의 또 다른 풀 항목일 뿐(파일·인라인·나라장터가 같은 포트).
2. **매트릭스 실행.** 대시보드에서 **잡(=템플릿+매핑) 다중선택 × 데이터 겨눔 → 일괄 생성**.
   `generate_batch`(1 템플릿 × N행)의 **M 잡 × K 데이터** 확장. 교차 충돌은 파일명 패턴에
   템플릿명 토큰을 넣어 방어(`OutputNamer` 연번·충돌 접미사 재사용).

3. **어휘 일괄 매핑.** **공유 소스 어휘를 1회 선언** → 각 템플릿 필드에 **이름 교집합으로
   투영**. 템플릿 필드는 어휘의 부분집합이라는 실사용 구조를 활용한다(코퍼스 실증:
   `tests/corpus/scenario` 의 입찰공고서 22필드 ⊇ 구매요청서 10필드 — 같은 CSV·나라장터
   소스가 두 템플릿을 채운다). 어휘에 없는 템플릿 필드는 조용히 비지 않고 **시끄럽게 미매핑
   신고**(ADR D 준용).

**UI 피드백 레이블 결정(2026-07-14, #14).** 한 작업×N행과 M작업×공통 데이터는 실제 생성
축이 달라 둘 다 유지한다. 전자는 `이 작업 실행`/`작업 1개로 데이터 각 행마다 문서 1건`,
후자는 `같은 데이터로 여러 작업 실행`/`선택한 작업 여러 개에 같은 데이터 적용`으로 부른다.
포괄적인 `일괄 실행` 한 단어로 두 기능을 가리키지 않는다.

> **뒤집힘(2026-07-18, F9).** J2 매트릭스 실행은 제거됐다 — 실무 실사용 부재("사실상 안
> 쓴다") + 축의 직교성이 사용자 머릿속에 "실행이 두 종류"라는 부담을 심어 데이터 모델을
> 오염시키고 대시보드 카피를 부풀렸다(completeness ≠ UI 자리값). 위 서사는 당시 논거의
> 기록으로 보존한다. 상세: `UX_FINDINGS_101_WALKTHROUGH.md` F9.

**명시성 게이트 보존(ADR D).** 일괄 매핑은 *편의*지 자동확정이 아니다 — 사람은 **어휘→정준
필드** 매핑을 1회 확정하고, 템플릿 투영은 **순수 이름 교집합**(새로운 조용한 바인딩을 만들지
않음)이다. 투영에서 빠진 템플릿 필드는 D의 "미매칭 hard-stop"으로 loud하게 남는다. 따라서
"묻고 확정하게 하라, 아니면 시끄럽게"(`confirm-or-alarm`)는 강화되지 약화되지 않는다.

**ADR A와의 관계 — 개정, 긴장을 정직하게 노출.** A의 정당화는 "사용자가 같은 `{템플릿+매핑}`을
많이 재사용한다 → Job 레지스트리가 진짜 자산이라 기본 진입"이었다. 3풀 분리는 **재사용 단위를
더 잘게 쪼갠다** — 이득은 조합 자유(한 데이터로 여러 양식, 한 어휘로 여러 템플릿), 위험은
멘탈모델이 풀 3개로 분산돼 A의 재사용-앵커가 흐려지는 것(A 경첩의 반대 압력). 대안 프레이밍:
**잡 = 풀 위의 저장된 조합(포인터)**, 매트릭스 실행 = 저장 없는 일회 조합. 진입은 여전히 자산
목록(A 유지)이되 자산이 3종이 된다.

**단계(각 단계 독립 출시 가능 — 전부 착지: 1=J1 `89e9cd7`, 2=J2 `7bbffd1`, 3=J3 `cac94c7`).**
1. 데이터 풀 + txt 인라인 입력 — 저위험, **H 후속을 닫는다**. 마법사에서 데이터 스텝 제거.
2. 매트릭스 실행 — 대시보드 잡 다중선택 × 데이터 1개(→ 이후 데이터도 복수).
3. 어휘 일괄 매핑 워크벤치 — 공유 어휘 선언 + 템플릿별 투영(게이트 보존).

**트레이드오프.** 풀 3개 표면은 선형 마법사보다 무겁고, 잘못하면 **"능력 없는 슬롯 발명"**
(핸드오프 관통 경고)의 재발 위험. 방어: 각 풀은 **실재 자산만**(잡·매핑 JSON·데이터소스),
매트릭스는 **기존 `generate_batch` 루프**, 어휘 매핑은 **코퍼스로 실증된 부분집합 투영**이라
발명이 아니라 이미 도는 코어의 표면화다.

**출처/실증.** `tests/corpus/scenario`(2026-07-12) — 22필드 조달 어휘를 공유하는 템플릿
2종(hwpx)·기안 2종(txt)과 소스 3형식(CSV·xlsx·나라장터 JSON)을 e2e로 물려, **한 소스가 여러
템플릿을 부분집합 투영으로 채움**을 헤드리스로 실증(`tests/test_scenario_e2e.py`). 이 ADR의
어휘-투영 주장을 뒷받침하는 실데이터 근거.

**코어 정합 확인(2026-07-12 토의).** 위 어휘-투영 전제를 **코드로 검증**했다. (a) alias는 이미
`FieldMapping.source`(영문키 ≠ 한글필드) + `suggest_mappings` 퍼지(`similarity`)로 동작 —
나라장터 응답에 대고 **22필드 중 12개 자동초안**, 나머지 10개는 소스 부재로 미매핑(loud).
(b) `MappingProfile.apply` 는 **템플릿-불가지**로 superset 를 방출하고 `engine.generate`
(`engine.py:42` 빈값 스킵)가 이름 교집합만 채운다 — 즉 **부분집합 투영이 코어에선 이미 된다**.
막힌 건 위저드가 프로파일을 한 템플릿 스키마에 시드(`MappingModel.from_suggestions(wiz.schema,
…)`)하는 **저작 표면뿐**이다. → "1:1은 UI 표면 인공물"의 코드 확증. 축 2의 작업 범위가 좁혀진다:
`apply`/`engine` 무변경, **위저드의 시드 소스를 "한 템플릿 스키마" → "공유 어휘"로 교체**.

**하위결정(① 착지 — V1 `4eb6ad0`) — 소스 어휘 소유권(NARA_ALIASES 탈하드코딩).** 종전 `core/mapping.py` 의
`NARA_ALIASES`(36쌍)는 두 냄새를 겹쳤다: (1) 범용 코어가 특정 API 어휘를 품음(README
"core = 제품 로직 없음" 위반), (2) GUI 가 **전 소스에** nara 어휘를 기본 퍼지 타겟으로 깖
(`from_suggestions` 기본 인자) — alias 는 전역 기본이 아니라 *선택된 소스*가 소유해야 한다.
이 사전은 실은 "나라장터 소스키 → 조달 어휘" **프로토 프로파일 = 축 2의 구체 인스턴스**라,
제거는 삭제가 아니라 **승격**이었다. 닫은 방향: **어휘는 소스가 선언**(`NaraStdDataSource.
field_labels()`), **코어는 어휘-불가지**(`suggest_mappings` 는 이미 `aliases` 를 범용 인자로
받음), **풀에서 일반화**. 순서 — ① (UI 무관·소규모) nara 어휘를 코어 밖(소스)으로 재배치,
동작 무변 — **완료(V1 `4eb6ad0`)**: 어휘는 `data/nara.py` 의 `NaraStdDataSource.field_labels()`
가 소유하고 코어에 `NARA_ALIASES` 는 더 없다(`from_suggestions` 의 nara 기본 인자도 제거,
`aliases=None`) → ② (풀과 함께) 소스-선언 어휘 + 저장 프로파일 수렴, "자동제안" = "키셋 맞는 풀
프로파일 재사용, 퍼지는 콜드스타트 폴백". **경고**: 퍼지 `suggest_mappings`/`similarity` 는
콜드스타트·필드 드리프트용으로 **유지**(생짜 삭제 금지); B-3b(나라장터 제품화)가 이 자동제안에
기대므로 **대체(소스 선언)를 붙인 뒤** 옮겨 UX 회귀를 막는다.

**축 확정(2026-07-12 · 도메인 답변).** 남은 J-축 두 도메인 질문에 사용자가 답해 축 1·3과 매핑
모델을 닫는다.
- **데이터 수명 → durable 데이터 풀(참조).** 데이터셋 수명 = 계약 사이클(발주~대금지급),
  **최소 60일·최대 1~2년**(계약기간 의존) 뒤 아카이브. 세션-일회가 아니므로 **recents 로 부족**
  → 데이터는 **durable 풀 항목**이되 **스냅샷이 아니라 연결/참조**(엑셀 경로·나라 쿼리)로 저장하고
  실행 때 재읽기("싱크"). 사이클 종료 = **보관** 액션(add/delete 아닌 archive — 실행 후보에서만
  제외, 참조 보존). *(#5: 한때 보관 위에 '은퇴' 3번째 상태가 있었으나 행동 차이가 없어 보관으로
  정준 병합 — 2상태 활성/보관.)* "데이터·행 미저장" 불변식 유지(포인터만 저장).
- **긴 수명은 조인이 아니라 ADR G 로 흡수 — 조인 파킹 유지.** 사이클 동안 데이터가 단계적으로
  자라도(낙찰자·계약일·검수일·지급일), 시간축 다단은 **누적 치환(ADR G)** + **출력 문서 = 상태
  원장**으로 처리한다. 따라서 **run/조합은 일회(저장 안 함)** — 별도 "케이스" 객체 발명 불필요.
  **인앱 조인/조립은 ADR K 로 분리·착지**(Power-Query식 파이프라인, 2026-07-12 착수 확정 →
  KA·KB 병합 — 최소 표면 GUI 쿼리빌더 실수요). 마스터시트 vs 계약별 시트 granularity·batch-cumulative
  파일키 매칭은 여전히 실데이터 landing 대상이나, K 착지는 이에 종속되지 않았다.
- **매핑 = 공유 베이스 + 인별 오버레이(개조).** 인별 재작성은 공수 과다 → **공유 베이스
  프로파일(풀)** + **잡/사용자 로컬 오버레이**(변경 행만 덮는 sparse 프로파일; `apply_profile`
  이 이미 오버레이 적용 기제). 베이스 편집은 **loud 전파 경고**("이 매핑을 잡 N개가 참조 —
  오버레이 안 한 행이 바뀝니다", ADR G 겹침 경고 패턴 재사용), 오버레이는 **격리**. 게이트
  (ADR D): 오버레이가 바꾼 행은 그 사람이 **재확정**(un-confirm→confirm) — 공유라고 자동확정 없음.

  > **뒤집힘(2026-07-18, F22).** 공유 베이스 프로파일(J3 이 축)은 제거됐다 — 작업이
  > 매핑을 자족 저장·복원하는 현 체계에서 별도 저장 개념은 배울 규칙만 늘렸고,
  > `base_mapping_name` 전파는 경고뿐 자동 재투영이 없어 "공유"의 실익조차 없었다
  > (matrix 보다 completeness 논거가 약함). 재사용은 홈 **「작업 복제」** 한 동선으로
  > 수렴. 위 서사는 당시 논거의 기록으로 보존한다. 상세: `UX_FINDINGS_101_WALKTHROUGH.md` F22.
- **축 1 착지(한 겹 vs 두 겹).** 조합은 **한 겹**(일회 실행·저장 안 함; 상태는 데이터셋 풀 +
  출력 원장에 산다). 단 **durable 프리미티브는 셋** — 잡(T+M) · 데이터셋(참조) · 매핑 베이스.
  즉 "두 겹(재사용 데이터-뷰 계층)"이 아니라 **평평한 3-풀 + 일회 조합**. A(진입=자산 목록) 유지,
  자산이 3종이 된다.
- **저장 위치(축 3 닫음).** 매핑 베이스·데이터셋 참조 모두 **홈 레지스트리**(`~/.hwpxfiller/`
  하위, 잡·txt 템플릿과 동일 관례; `HWPXFILLER_HOME` 재지정). 오래 걸려 있던 **매핑 프로파일
  레지스트리 미결을 이 풀이 흡수**. 데이터셋은 참조 + 아카이브 상태를 가진 durable 항목.
- **축 2 착지(출력측 어휘, 닫음).** 정준 이름셋 = 사람이 확정·소유하는 **공유 베이스 매핑의
  필드 집합**(별도 vocab 아티팩트 불필요). 템플릿 합집합은 저작 캔버스 **프리필 편의**(입력)일
  뿐 소유물이 아니다. 파편화 안전망 = 기존 **합성 게이트(ADR D 미매칭 loud)** — 단 종전 `run_state`
  가 드리프트-유입 필드를 "의도적 공란"으로 오라벨하던 구멍은 **ADR L(생성 원장)이 상태 구별로
  닫았다**(L1 `2188d4d`). `similarity` 는 제안·진단으로만 잔류. 명시 `--vocab` 은 조직 표준 freeze 옵션(강제 아님).

## L — 생성 원장(프로버넌스) + 워크플로 드리프트 감지 · **착지(2026-07-12·L1·L2 병합)**

**상태.** **착지** — L1(공란 선언·전건 커버·드리프트 게이트, `2188d4d`)·L2(원장 export·소스
프로파일링·되읽기 증거, `367e949`) 전부 master 병합(`core/fill_ledger.py`·`core/source_profile.py`).
축 3(드리프트 기전·트리거)·축 1(원장 척추)·축 4(산출) 닫힘, 축 2(프로파일링 깊이)는 방향 닫힘 후
수치도 구현으로 확정 — 아래 "축 확정"·"잔여" 블록. J와 **직교**(풀 유무 무관, 파이프라인에 흐름)하되 J가 증폭한다(다템플릿·공유매핑
에서 드리프트 원천이 셋으로 늘어 원장의 필요가 커짐). 지난 논의의 "생성 전 드리프트 리포트"
요구를 이 원장의 **한 단면**으로 흡수한다.

**문제(피드백).** 사람은 생성 전에 "매핑이 되어 있다"(구조)를 넘어 **헤더의 실제 데이터형·들어갈
결과값**을 보고, 생성 후엔 **들어갔음을(원하면) 추적**하고 싶다. 그리고 템플릿이든 정준 매핑셋이든
**드리프트가 워크플로 중 즉시** 감지돼야 한다.

**핵심 관찰 — 추출엔 이미 원장이 있다.** `text_extract.py` 의 **coverage ledger**("미모델링 구조
시끄럽게 신고")가 *추출* 측 프로버넌스다. 요구는 그 **생성 측 대칭 = 생성 원장(fill ledger)** —
한 필드가 소스→변환→누름틀→문서로 흐르는 계보를 필드마다 좇는다.

**결정(제안→착지) — 생성 원장.** 필드당 한 행: {누름틀, 소스 출처, **소스 실제형(샘플)**, 변환·표시형,
**결과값(dry-run)**, 상태, (사후)주입 ✓/✗}.
- **생성 전** = dry-run 매니페스트(무엇이 어떻게 들어갈지) → 사람 확정 지점.
- **생성 후** = 원장 닫음(`GenerateResult.applied`/`unmatched` 대조) → "들어갔음" 증거. **opt-in
  export/첨부**.
- **드리프트** = 매 실행 시 세 이음새 diff를 **상태값**으로 loud: ① 소스⊖매핑(컬럼 소실·형 변화),
  ② 매핑⊖템플릿, ③ 정준 철자. **의도적 공란 ≠ 드리프트 유입**을 구별(축 2가 지목한 오라벨 해소).

**있는 것 / 새것(코드).** 재료 대부분 실재 — 결과값 `RunRequest.mapped_records`, 템플릿·매핑 필드
`required_fields`/`job.template_fields`(run_state가 이미 diff, 라벨만 뭉갬), 사후 `GenerateResult`,
표시형 `format_engine`. **새것 셋 — 전부 착지**: (1) **소스-측 프로파일링**(샘플+잠정 타입) =
**D-8의 소스 측**, 파킹 사유가 "책상 위 추정"이었고 실데이터 확보로 해동 →
`core/source_profile.py`(`FieldProfile`·`profile_fields`, 서술적·degrade·주장 아님);
(2) 통합 원장 dataclass(추출 ledger의 생성 짝) → `core/fill_ledger.py` 의 `FillLedger`·`LedgerRow`;
(3) 이음새 드리프트 상태 라벨링 → `TemplateStructureDrift`(L1 `2188d4d`·L2 `367e949`).

**경계(원칙 보존).**
- **값 미리보기 ≠ HWPX 렌더(ADR C 불변).** 원장은 *필드 값*을 보이지 문서를 렌더하지 않는다 —
  한글이 권위 렌더러라는 C를 안 건드림. "인앱 미리보기" 발명 방지선.
- **소스 프로파일 = 서술적·degrade**(D-8/표시형 아크 착지점) — 타입 *주장* 아닌 *제안*, 모르면
  샘플만.
- **추적 opt-in**("원하면") — 강제 감사로그 아님(배치당 원장 export). 도메인 정당성: 공고서 =
  법적효력 이상 사분면이라 고위험 문서엔 계보 실익, 나머진 끔.
- **비-목표 유지** — 프로버넌스는 "값 치환의 확장"의 메타데이터지 문서 조립·감사 서브시스템이
  아님(원장 export 수준에 못박음).

**연결.** coverage ledger(거울·정박)·ADR C(값≠렌더 경계)·D-8(소스 프로파일 해동)·ADR B/E(채움/
빈칸/미입력 상태를 원장이 일반화)·로드맵 ⑤ 출력검증(`MISSING_MARKER` grep을 원장이 형식화)·
ADR J(다템플릿이 드리프트 원천 증폭).

**축 확정(2026-07-12 · 이어진 토의 — 오라벨 구멍을 코드로 해부해 닫음; L1 `2188d4d`로 구현 착지).**
당시 `run_state.field_states`(현행 `run_state.py:317`)가 `_intentional_blanks`(당시 = 템플릿
누름틀 ⊖ 매핑 **추론**; 현행 후신 = 선언 기반 `blank_fields`, `run_state.py:299`)를 전부
`blank`(의도적)로 라벨하는 게 오라벨의 실측 지점이었다 —
한글 재편집으로 유입된 누름틀이 "의도적 공란"이라는 **안심 라벨로 조용히** 분류됐다(경보 자체가 안
울림). 소스측은 `source_report`(`job.py:215` = `validate(source_keys, data)`)가 `missing_columns`로
loud하나 **템플릿측엔 대칭 계약이 없었다**(`output_report`가 `job.py:224`에서 매핑을 *자기 자신*과
검증 — 재확정 잡음 회피는 옳으나 템플릿 성장/축소에 눈을 감음). 이 비대칭을 다음으로 닫았다
(L1 구현으로 실코드화 — `_intentional_blanks`는 이제 선언 기반 `blank_fields`, 대칭차는
`run_state.structure_drift`/`TemplateStructureDrift`):

- **의도적 공란 = 매핑의 일급 선언(추론 폐기).** 의도는 **템플릿이 아니라 매핑에서만** 선언된다 —
  덩그러니 놓인 미매핑 템플릿 토큰은 정상 상태가 아니다. `FieldMapping`에 명시적 공란 항목(가산)을
  두어 **매핑이 템플릿 누름틀을 전건 커버**(각 토큰 = {소스 매핑 | 명시적 공란}, ADR D `is_complete`
  전건 확정의 연장). `_intentional_blanks`가 추론(차집합)→선언(매핑 항목)으로 전환. 엔진 동작 불변.
- **드리프트 = 매핑 커버 집합과의 대칭차(스냅샷 불요).** 전건 커버가 성립하면 **매핑 커버 집합이 곧
  기준선** → 드리프트 = (현재 템플릿 누름틀) △ (매핑 커버). 양방향 loud: 미매핑 토큰 유입 /
  매핑되던 토큰 소멸 — 후자는 현재 `GenerateResult.unmatched` 사후·미표면 **침묵 구멍**(engine.py:62)을
  닫는다. 별도 이력/스냅샷 저장물 없음(매핑이 계약).
- **구조 드리프트 ≠ 값 드리프트(ADR E 보존).** 값 수준 공란(레코드별 빈 값)은 **ADR E ack 유지**
  (강제 상호작용 후 진행 — 레코드마다 정당). **토큰 집합 변화(구조)는 ack 대상 아님** — 계약 수리지
  레코드 판단이 아니다. 이 분리를 명시하지 않으면 "무시 없음"이 값 공란까지 하드락해 ADR E와 충돌.
- **구조 드리프트 차등 처방(J1 정합) — "무시 없음"은 템플릿-구조 한정.**
  - **템플릿-구조 드리프트**(Job의 durable 구성요소 `template_path`) → **하드게이트·무시 없음** →
    매핑 재확정으로 전건 커버 회복. `confirm-or-alarm` "허용 전이는 확정요구·실패표시 둘뿐, 조용한
    추측 금지"의 순수 적용. C3 위저드 PARTIAL 하드게이트의 매핑-드리프트층 대응.
  - **소스-구조 드리프트**(J1 데이터 풀 = 실행마다 재읽기하는 **휘발성 참조**) → loud but 실행-스코프,
    처방 = 매핑 수리가 아니라 **데이터 포인터 교정** or 문서 내 `MISSING_MARKER`. **J1의
    durable(템플릿)/휘발성(데이터) 비대칭이 이 차등을 정당화** — "무시 없음"을 소스측에 확장하면 정상
    변동하는 풀 소스가 실행을 막아 J1과 충돌.
- **원장 export = 포인터-온리 + N1 redaction.** opt-in 사이드카(축4)는 나라 쿼리를 **박제하지 않고
  풀 항목을 포인터 참조**(J1 "포인터-온리, 비밀은 OS 저장소" 불변식 상속) + **N1 redaction 관통**
  (URL 내 ServiceKey 유출 방지 — 원장 export가 새 유출면).
- **전방(J3 대비) — 커버/드리프트는 유효 매핑에서 평가.** J3 매핑 = base ⊕ overlay 착지 시에도 안
  깨지도록 전건 커버·대칭차는 **합성된 effective mapping** 기준으로 평가한다(지금 명시 → J3 재작업 0).
- **사후 검증 실화(C1 착지).** `set_prev_output`(`run_state.py:218`)가 "값 읽기 API 부재"로 파킹했던
  값 수준 검사가 **C1 `read_field` 착지(커밋 `b76e247`)로 해동** → 원장 사후 닫음이
  `GenerateResult.applied`(엔진의 주장)를 넘어 **생성물 실값 되읽기(증거)**로 검증한다 — L2
  `verify_output`(`core/fill_ledger.py`)이 구현. ADR G 값겹침 격상은 같은 API를 공유하되 **미착**
  (openQuestions G 후속).

## K — 조립 소스 / 데이터 파이프라인 · **착지(2026-07-12·KA·KB 병합)**

**상태.** **착지** — 목표 형태 확정에서 구현 착수로 승격(2026-07-12, 사용자 판단) 후 **KA·KB
전부 master 병합**: KA 코어 = `data/pipeline.py`(`AssemblyEngine` 이음새·`PipelineSource`·재귀
복원, `1fe0b48`), KB GUI = `gui/pipeline_builder.py` 비주얼 쿼리빌더가 J1 풀 표면
`gui/dataset_pool_panel.py`에 통합(`71c0041`; 적대 리뷰 4결함 봉합 `d511944`).
종전 "수요 파킹"의 유일한 파킹 근거는 "수요뿐"이었는데, 제품 소유자가 **"최소 표면 GUI
쿼리빌더는 제품 설계상 필수이고 실수요가 있다"**고 판단해 그 게이트를 충족 선언했다. J에서
조인/조립을 떼어 별도로 둔다: **J = 다대다 오케스트레이션**(풀·매트릭스·어휘), **K = 소스
조립**(여러 원천 → 하나의 `DataSource`). ADR J §축확정의 "인앱 조인 파킹"은 이 승격으로 닫힌다.

**결정.** "여러 데이터 → 한 순간 한 문서"(동시 결합)를, **사용자가 저작·미리보기하는
Power-Query*식* 파이프라인**으로 착지한다 — `DataSource`를 *생산*하는 층(이음새 아래),
**추론 엔진 아님**.

**착수 시 4결정(2026-07-12 확정) — "최소 표면" 판단이 셋을 일관되게 정렬한다.** 종전 트레이드오프
절의 3중 방어(수요게이트·최소노출·어댑터이음새) 중 **수요게이트가 걷혔으므로, "최소 노출"이
남은 유일한 가드레일**이다("최소한"의 구체값이 곧 방어선). 그 판단이 아래로 캐스케이드한다:
1. **엔진 = petl/agate(경량 순수파이썬), DuckDB는 후일 교체.** DuckDB 선호의 유일 근거는
   "SQL=스텝 언어"인데, 그 이점은 **raw SQL을 노출할 때만** 값을 낸다. 최소 표면(비주얼 위젯만,
   SQL 미노출)에선 DuckDB는 안 쓰는 무게 + PyInstaller 번들 압박(K1). 경량 엔진이 K1 번들
   마찰 0. **`AssemblyEngine` 프로토콜 + 교체점**이 후일 "고급 편집=raw SQL" 실수요 시 DuckDB
   교체를 값싸게 보장(`format_engine.ENGINE` 선례).
2. **자산 종류 = 데이터셋 풀 하위종류(`kind="pipeline"`), 제4 durable 자산 아님.** J1 구현
   (`core/dataset_pool.py`)이 이미 `DatasetPoolItem{kind,opts}`를 kind-불가지로 저장 → 파이프라인은
   4번째 상자 없이 흡수. 홈 레지스트리 자산 3종 유지(잡·데이터셋참조·매핑베이스), A 경첩(재사용률)
   인지부하 최소화. 종전 본문의 "제4 durable 자산" 서술을 **철회**한다.
3. **v1 스텝 집합 = merge(키 조인) + append(union)** 두 코어 위젯 + WYSIWYG 미리보기.
   filter(행)·select/rename(열)은 **'고급 편집' 뒤로 접음**(기본 노출 시 ETL 팽창 신호;
   열 이름 재작성은 매핑 베이스 층과 혼동 위험). raw SQL/스텝 편집은 **v1 밖**(열리는 순간 DuckDB
   필요 + 범용 ETL 문). **falsifiable 가드: 스텝 집합이 이 넷을 넘어 자라면 = 슬롯 발명 위험
   실현 신호 → 정지.**
4. **구현 = 2유닛 분할(이음새가 분할선).**
   - **K-코어(격리 리프, 자율 에이전트 후보).** 신규 `core/assembly.py`(or `data/pipeline.py`):
     `AssemblyEngine` 프로토콜 + petl/agate 구현 + `PipelineSource(DataSource)`. `data/factory.py`:
     `make_source`에 `kind=="pipeline"` 분기 + `source_from_pool_item` **재귀 복원**. `dataset_pool.py`
     **무접촉**(kind 불가지). 파일 서로소 → P1/K1/C1 몰드로 병렬·게이트 병합. 나라 코어가 CLI로
     먼저 랜딩한 전례와 동형(헤드리스 `DataSource`부터 사용 가능).
   - **K-GUI(직렬, 웨이브 2/3).** 비주얼 쿼리빌더를 J1 데이터셋 풀 표면
     (`gui/dataset_pool_panel.py`·`dataset_pool_state.py`, 경우에 따라 `gui/home.py`)에 끼움 —
     `엑셀 | 나라 | 파이프라인` 선택지 추가. J1/C4/N2 핫스팟 뒤 직렬.
   - **선확정 계약(코어가 GUI 무지로 격리 개발해도 attach 무충돌).** 풀 항목 스키마
     `{kind:"pipeline", opts:{sources:[<sub-ref>...], steps:[<recipe>]}}` + `source_from_pool_item`이
     `kind=="pipeline"`에서 **각 sub-source를 다시 `source_from_pool_item`으로 재귀 복원**. 핵심:
     나라 sub-source가 이 재귀를 타면 기존 **SecretStore 키 주입 경로**(`factory.py` nara 분기)를
     그대로 상속 → 보안 불변식이 공짜(make_source 직결 우회 금지). **excel 항목은 확정 시트를
     opts에 임베드**(T2, ADR M): `{kind:"excel", opts:{path, [sheet], [header_row]}}` —
     최상위 항목이든 파이프라인 sub-ref든 복원은 공통 폴스루(`make_source(kind, **opts)`)라
     **재질문 없이 등록 시점에 확정된 그 시트를 다시 겨눈다**. 참조는 재현에 충분해야 한다 —
     시트가 opts 밖이면 복원이 첫 시트를 조용히 추측하는 구멍이 재발한다.

- **가공방식 레시피이지 데이터가 아니다(축 1 정합).** 파이프라인은 **저장되는 데이터 *가공방식*
  레시피**다 — 매핑 베이스가 *값* 가공방식(1→1·N→1·서식)이라면, 이건 *소스 집합* 가공방식
  (merge/append/filter/select). **둘 다 데이터를 스냅샷하지 않고 참조만**(실행 때 재읽어 레시피
  재실행). 따라서 한 겹 불변식("데이터·행 미저장, 포인터만")을 그대로 지킨다 — 단 **별도의 제4
  자산이 아니라 데이터셋 풀 하위종류(`kind="pipeline"`)로 흡수**된다(결정2; 홈 레지스트리 자산 3종
  유지: 잡·데이터셋참조·매핑베이스).
- **층위 — 이음새 아래의 소스 생산자.** 조립 결과는 `DataSource`(`data/base.py` 프로토콜:
  `records()`/`fields()`)를 만족해 **다운스트림(매핑·엔진·작업·실행)엔 단일 소스로 보인다**.
  `data/factory.py`의 새 `kind`("pipeline")로 노출(현 factory의 미래 `"cumulative"` 확장점과
  같은 자리). **필드 변환(`mapping.apply_transform`)과 다른 층** — 저건 레코드 *내부* 값, 이건
  레코드 *집합* 조립. 혼동 금지.
- **재발명 금지 = 외부 엔진 래핑, 어댑터 이음새 뒤.** `core/format_engine.py`의 `FormatEngine`
  프로토콜 + `ENGINE` 교체점 선례 그대로 `AssemblyEngine` 프로토콜 + 교체점을 둔다 — 코어는
  이음새에만 의존, 무거운 라이브러리는 착지 시에만·교체 가능하게. Power Query M 엔진은 임베드
  불가라 *철학만* 차용하고 엔진은 기존 테이블 라이브러리. 후보(결정1로 v1 확정, 교체점 뒤):
  **petl/agate**(v1 선택 — 경량 순수파이썬, 최소 표면에 K1 번들 마찰 0) · DuckDB(후일 교체 —
  SQL이 곧 명시·미리보기·기계생성 가능한 스텝 언어이나 **raw SQL 노출 시에만** 값, 최소 표면
  v1엔 안 쓰는 무게; 자기완결 단일 바이너리) · pandas/polars(표준이나 무거움, PyInstaller 번들
  압박).
- **필요한 만큼만 노출.** 범용 ETL 아님 — 실제 조립 수요에 맞춘 curated 스텝(키 merge·append·
  filter·select/rename). 스코프 가드: "값 치환의 확장을 위한 소스 조립"이지 데이터웨어하우스
  아님. 간단 기본값 + '고급 편집'(raw SQL/스텝) + degrade(= format_engine '직접 입력…' 동형).
- **작성 자동화 = 제안 + 게이트, D-8 함정 계승.** 공유키 감지 → merge 스텝 *제안*은 **제안 전용·
  미리보기·사람 확정(ADR D)**. 추천 품질은 **실데이터로만 검증**(파킹된 D-8 타입 추천기와 동일
  근거 — 책상 위 추정 금지). 추측한 조인 자동 실행 금지.
- **UnivContractor 교훈 착지.** #2: merge 키 = **표시축(preset) 아닌 구조축(schema)** — 실제
  공유 컬럼·사용자 명시·미리보기, preset 이름 휴리스틱 금지. #3: 조립 파이프라인은 모든
  소비자가 **공유**(표시/생성 divergence 없음).

**착수 근거 — 철학이 아니라 관측된 수요.** Power-Query식 명시 저작은 "책상 위 추정 금지"를 이미
해소한다(추정 주체가 도구가 아니라 사람). 유일한 파킹 근거였던 **수요**는 단언이 아니라
**이전 운용에서 실 2+소스 워크플로로 검증됐다**(관측된 사실 — 최소 표면 GUI 쿼리빌더 = 제품 필수,
2026-07-12 제품 소유자 확인). 종전의 "나라 세부 op·ERP API가 나타날 때"라는 *접근-선행형* 게이트는
런타임 관측 장치가 없어(스톱갭 (A) 엑셀 임시배관이 수요 신호를 조용히 흡수) 향후 수요를 스스로
falsify하지 못했을 뿐, **수요 자체는 이미 운용 관측으로 성립**한다 — 그래서 접근-선행형 게이트를
**운용 검증에 근거한 제품 판단으로 대체**한다. batch-cumulative 파일키
매칭·데이터 granularity는 여전히 실데이터 landing 대상이나, K 코어 착수는 이에 종속되지 않았다.

**트레이드오프.** 파이프라인 표면은 무거운 능력이라 "능력 없는 슬롯 발명"(핸드오프 관통 경고)의
재발 위험이 가장 크다. **수요 게이트가 걷혔으므로 방어 하중은 남은 둘에 실린다**: **최소 노출**
(범용 ETL 금지 — v1 스텝셋 merge+append로 고정, 위 falsifiable 가드) + **어댑터 이음새**(코어가
라이브러리를 붙들지 않음 — petl→DuckDB 교체점). 최소 표면 + 실수요라는 조건 자체가 이 위험이
*실현되는 조건*(범용 ETL로의 팽창)을 배제한다.

## M — 파일 겨눔 소결정: 시트 확정(T2) · 마지막 디렉터리(T3) · **착지(2026-07-14)**

**문제.** (1) 다중 시트 통합문서를 겨누면 `ExcelDataSource`가 **첫 시트를 조용히 추측**해
읽었다 — 사용자가 둘째 시트를 의도했을 때 잘못된 데이터로 문서가 생성되는 침묵 오류
(confirm-or-alarm 위반의 전형). (2) 파일 다이얼로그가 매번 OS 기본 위치에서 열려 반복
작업마다 같은 폴더를 다시 찾아 들어가야 했다.

**결정 1 — 시트 확정(confirm-or-alarm의 순수 적용).** 다중 시트(2개 이상)면 **확정을
요구**한다: `gui/view_helpers.py` `ask_sheet_choice`가 `QInputDialog.getItem`으로 묻고
(풀 선택 getItem 선례 동형), 5표면(위저드·실행·매트릭스·txt·풀 등록)이 이 한 헬퍼를
공유한다. 원칙 적용의 세부:

- **취소 = 겨눔 전체 중단.** "허용 전이는 확정요구·실패표시 둘뿐" — 취소를 첫 시트
  기본값으로 강등하면 그게 곧 조용한 추측이다. 이전 겨눔 상태는 보존된다(풀 등록에선
  추측 참조가 조용히 저장되는 것을 막는다).
- **질문 생략은 추측이 아니다.** CSV(시트 개념 없음)·단일 시트는 **물을 것이 없어** 묻지
  않는다 — 생략 판정의 단일 출처는 T1 `data/excel.py` `sheet_overview`(빈 목록 = CSV,
  길이 1 = 유일 시트). 여기서 다이얼로그를 강행하면 확인 피로(ADR E 계보의 습관화 위험)만
  산다. 시트 열거 실패(손상 파일)는 로드 실패와 같은 경로로 **시끄럽게** 알린다.
- **확정은 재진술된다.** 어느 시트가 겨눠졌는지 침묵하지 않는다 — 파일 라벨
  `[시트: X]`·위저드 요약 `— 시트: X`·풀 카드 `· 시트 X`(문구는
  [UI_VOCABULARY.md](UI_VOCABULARY.md) §6).
- **확정은 하류 계약에 관통한다.** ① 매핑 초안 캐시 키(`wizard.py` RC-09 키)에 시트를
  포함 — 같은 경로·같은 헤더의 **다른 시트** 재선택이 옛 초안으로 조용히 구동되지 않는다.
  ② 풀 항목 `opts`에 시트 임베드(ADR K 선확정 계약 병기 참조) — 복원이 재질문 없이 같은
  시트를 겨눈다. ③ VM은 확정된 값만 수취: `load_data(path, sheet=None)` 키워드 확장
  ([UI_CONTRACT.md](UI_CONTRACT.md) 데이터 겨눔 콜백, 다이얼로그는 링2 —
  [ARCH_UI_SEPARATION.md](ARCH_UI_SEPARATION.md)).

**결정 2 — 용도별 마지막 디렉터리(편의-확정 경계의 반대편).** `view_helpers.py`
`last_dir`/`save_last_dir`가 파일 다이얼로그 **시작 디렉터리**를 용도별
키(template/data/mapping/prev_doc/output/txt_save/library)로 기억한다. 시트 확정과 정반대
규율이 옳다는 게 이 결정의 요점이다:

- **편의이지 확정 대상이 아니다 — 조용한 폴백(ST-11 지오메트리 지속과 동급).** 미저장·
  손상 값·삭제된 디렉터리는 예외 없이 빈 문자열(OS 기본 시작 위치)로 강등한다. 시작
  위치가 틀려도 사용자 과업(파일 선택)은 훼손되지 않으므로 여기에 경보를 달면 잡음이다 —
  confirm-or-alarm은 **불확실성이 결과물을 오염시킬 수 있는 전이**에만 확정을 요구한다.
- **성공 선택만 저장**(취소는 직전 기억 보존) · **부모 디렉터리만**(다음 열기가 직전 선택
  곁에서 열리게) · **파일명 프리필 금지**(시작 위치만 제공 — 이름은 항상 사용자 몫,
  저장 다이얼로그의 기존 `mapping_profile.json`·`기안.txt` 프리필은 제거).
- **용도 키 분리 근거:** 템플릿·데이터·출력 폴더는 통상 서로 다른 위치에 산다 — 단일
  전역 키면 마지막에 연 다이얼로그가 다른 용도의 기억을 밀어내 편의가 상쇄된다.

**경계.** 두 결정은 같은 표면(파일 다이얼로그 인접)에 있지만 반대 규율을 받는다: 시트
선택은 **데이터 정체성**(틀리면 잘못된 문서가 조용히 생성) → 확정 요구, 시작 디렉터리는
**순수 편의**(틀려도 클릭 몇 번) → 조용한 폴백. 이 경계 자체가 confirm-or-alarm의 적용
기준을 예시한다.

## 관통 원리

두 개의 앵커로 6개 결정이 닫혔다.

1. **불일치·누락을 UI 구조가 불가능하게 만들거나 시끄럽게 만든다** — 계보의 원죄(조용한 실패)를
   입력단(F)·확정단(D)·표현단(B)·표면화단(E)에서 각각 차단.
2. **권위 있는 렌더러를 흉내 내지 않는다** — 한글이 진실의 출력이므로 인앱 미리보기는 충실도
   위험(C). 통념("라이브 분할 미리보기가 표준")은 계보상 근거가 없다. 단 이 원리는 트랙에 따라
   반대 표면을 낳는다 — txt엔 외부 권위 렌더러가 없어 실시간 view가 곧 정답.
3. **무게 분할이 분기를 정한다** — 출력 표면을 건드리는 결정(C, E의 형태)은 트랙의 출력
   물리(외부 파일+권위 렌더러 vs 실시간 view+클립보드)가 달라 트랙별로 갈라진다. 공유 엔진(B
   표식·D 매핑·F 선택·서식)을 건드리는 결정은 두 트랙이 같은 매핑/서식 모델을 소비하므로 통일된다.

### 가산 필드 version-불변 규칙 (`core/job.py`)
`Job` 의 사용 메타 가산 필드(`last_run_at` 마지막 성공 실행 ISO·`tags` 작업 브라우저 축)는
**`version` 을 올리지 않는다** — `from_dict` 가 `.get` 기본값으로 하위호환을 지킨다. 엔진은
합성된 `mapping` 만 소비하고, 이 메타들은 "데이터·행 미포함" 불변식과 무관한 순수 메타다.
역방향도 대칭: 제거된 가산 필드(`base_mapping_name`, F22)는 미지 키로 무시된다 —
구 JSON 로드 무해, 재저장 시 키 자연 소멸.

### 앱 A — diff 리뷰어 뷰 계약 (`src/hwpxdiff/`)
> 앱 A(diff 리뷰어)는 앱 B와 **별도 제품**(읽기 도구)이라 본 문서(앱 B)와 스코프가 다르나,
> ARCH·CONTRACT 가 앱 A 를 스코프 밖으로 두어 이 뷰 결정의 수용 문서가 달리 없어 여기 보존한다.

**코어 계약** (`hwpxdiff/diff.py`, 완성·골든 고정 — UI 가 건드리지 않음):
- `diff_files(old, new) -> DiffResult` — 실코퍼스 ~30ms(동기 실행 충분; 대형 문서 워커 스레드는
  개선 후보 = GitHub #6).
- `DiffResult.rows` — **equal 포함 전문(全文) 대조 스트림**(`DocRow`; 변경 행 `seq`==`Change.seq`,
  equal 은 None). 골든 밖의 뷰 데이터 = 신구대비표의 원천.
- `DiffResult.change_items`·`render_html` 은 CLI(`diff --html`) 전용 유지 — GUI 는 안 씀.
- `KIND_LABELS`/`KIND_COLORS` — 종류(추가/삭제/변경/번호변경) 어휘·색 단일 출처.

**뷰 형태** (2026-07 사용자 피드백 확정 — 변경분-only 리포트 폐기):
- **전문 신구대비표** — 원문 전체를 좌(구판)/우(신판) 대조(`_render_doc_html`). 변경만 발라내면
  본문 맥락이 날아간다는 피드백의 산물. 변경 행은 구판 del/신판 ins 로 갈라 강조, equal 은 좌우 동일.
- **변경 리스트 = 네비게이션** — 문서상 인접·동종 변경을 그룹 1행으로(`diff.group_changes` →
  `DiffResult.change_groups`). 클릭 → `scrollToAnchor(chg-{첫 seq})`.
- **필터 = 종류 3종 고정**(추가/삭제/변경) + 번호변경 전용 토글(기본 접힘·개수 노출). 세분
  범주(숫자/조항…) 필터는 노이즈라 제거.
- **내보내기 없음**(「브라우저에서 열기」·「HTML 저장」 제거). 낱말 diff 파편화는 코어
  `diff.coalesce_word_ops` 가 완화 — GUI 전문 뷰와 CLI HTML 이 같은 성형 공유.

**발명 금지:** 읽기 도구다 — 문서 편집·주석·병합 금지(스코프 밖). diff 정밀도(표 제목 매칭·셀
재번호 인지)는 코어 파킹 항목이지 UI 일이 아니다.
